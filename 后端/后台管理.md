针对**后台管理后端服务**，我将其拆分为 **8个核心模块** 进行深度细化。后台管理后端是整个充电平台的"大脑"，支撑运营、财务、市场、管理层等多个角色的日常工作，覆盖站点运营、设备管理、订单管理、财务结算、营销管理、数据报表、系统配置、权限管理等全链路能力。

## 提示词 1：超级权限中心与多租户安全基座

> **定位**：后台管理的安全基石。支撑多角色、多组织、多租户的精细化权限管控，确保不同层级的运营人员只能看到和操作自己权限范围内的数据。

```text
请生成充电平台"后台管理后端"的"超级权限中心与多租户安全基座"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + Spring Security 6 + Sa-Token (或 Apache Shiro) + PostgreSQL + Redis + JWT。
设计目标：支撑 50+ 角色、1000+ 权限点、100+ 租户，实现菜单+按钮+数据+字段四级权限精细化管控。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、多租户架构设计】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 租户模型设计：
   - 租户类型：
     * 平台运营方（SuperTenant，管理整个平台）
     * 充电运营商（运营商 A、运营商 B，管理自己的站点和设备）
     * 物业公司/场地方（只读查看自己的站点数据和分成）
     * 政府监管部门（查看辖区内站点数据，只读权限）

   - 租户层级关系：
     * 平台运营方
       ├── 运营商 A
       │   ├── 区域分公司 A1
       │   └── 区域分公司 A2
       ├── 运营商 B
       │   └── 区域分公司 B1
       └── 物业合作方（只读）

2. 租户数据隔离策略：
   - 核心业务数据（站点、设备、订单、用户）：
     * 使用 tenant_id 字段隔离（共享数据库）
     * 通过 MyBatis-Plus TenantLineHandler 自动注入 SQL 条件
     * 所有查询自动追加 WHERE tenant_id = ?
   - 敏感数据（财务报表、核心配置）：
     * 使用 Schema 隔离（每个租户独立 Schema）
   - 超级管理员：
     * 可跨租户查看数据，需特殊权限 SUPER_ADMIN

3. 租户上下文管理：
   ```java
   // 使用 ScopedValue (Java 21) 存储租户上下文，兼容虚拟线程
   public class TenantContext {
       private static final ScopedValue TENANT = ScopedValue.newInstance();
       
       public static TenantInfo current() { return TENANT.get(); }
       
       public static void run(TenantInfo tenant, Runnable action) {
           ScopedValue.runWhere(TENANT, tenant, action);
       }
   }
   
   // 拦截器自动设置租户上下文
   @Component
   public class TenantInterceptor implements HandlerInterceptor {
       @Override
       public boolean preHandle(HttpServletRequest request, ...) {
           String tenantId = getTenantFromToken(request);
           TenantContext.run(new TenantInfo(tenantId), () -> {
               // 设置 MyBatis-Plus 租户条件
               TenantContextHolder.setTenantId(tenantId);
           });
           return true;
       }
   }
   ```

【二、RBAC 权限模型设计】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 权限模型结构：
   - 用户（User）←→ 角色（Role）←→ 权限（Permission）
   - 权限类型：
     * MENU（菜单权限，如"站点管理"菜单）
     * BUTTON（按钮权限，如"新增站点"按钮）
     * API（接口权限，如"POST /api/v1/admin/stations"）
     * DATA（数据权限，如"只能查看自己区域的站点"）

2. 预置角色体系（按组织层级）：

   【平台级角色】
   - 超级管理员（SuperAdmin）：拥有全部权限，可管理所有租户
   - 平台运营总监：管理所有运营商的运营数据
   - 平台财务：查看所有运营商的财务数据
   - 平台技术支持：管理设备、处理技术工单

   【运营商级角色】
   - 运营总监：管理本运营商全部业务
   - 区域经理：管理指定区域的站点和运维
   - 站点管理员：管理指定站点的日常运营
   - 财务人员：管理本运营商的财务、对账、发票
   - 客服人员：处理用户投诉、退款
   - 市场人员：管理营销活动、优惠券
   - 数据分析师：查看报表（只读）

   【物业/场地方角色】
   - 物业管理员：查看本站点的充电数据、分成收益
   - 物业财务：查看财务报表、发票

3. 权限粒度细化：

   菜单权限示例：
   - 站点管理（MENU）
     ├── 站点列表（MENU）
     │   ├── 查看站点详情（BUTTON）
     │   ├── 新增站点（BUTTON）
     │   ├── 编辑站点（BUTTON）
     │   ├── 删除站点（BUTTON）
     │   └── 导出站点（BUTTON）
     ├── 电价管理（MENU）
     │   ├── 查看电价（BUTTON）
     │   ├── 编辑电价（BUTTON）
     │   └── 审批电价（BUTTON，需审批权限）
     └── 站点地图（MENU）

   API 权限示例：
   - station:list → GET /api/v1/admin/stations
   - station:create → POST /api/v1/admin/stations
   - station:update → PUT /api/v1/admin/stations/{id}
   - station:delete → DELETE /api/v1/admin/stations/{id}
   - station:export → GET /api/v1/admin/stations/export

4. 数据权限设计（核心亮点）：

   数据权限类型：
   - 全部数据（ALL）：可查看所有数据
   - 本组织及下级（ORG_AND_CHILDREN）：可查看本组织和下级组织的数据
   - 仅本组织（ORG_ONLY）：仅查看本组织的数据
   - 自定义组织（CUSTOM_ORG）：指定可查看的组织列表
   - 仅本人（SELF_ONLY）：仅查看自己创建/负责的数据

   实现方式（MyBatis-Plus 拦截器）：
   ```java
   @Component
   public class DataPermissionInterceptor implements InnerInterceptor {
       @Override
       public void beforeQuery(Executor executor, MappedStatement ms, ...) {
           // 获取当前用户的数据权限配置
           DataPermissionConfig config = getDataPermissionConfig();
           
           // 根据配置动态拼接 SQL 条件
           if (config.getType() == DataPermissionType.ORG_ONLY) {
               // WHERE org_id = {currentOrgId}
               addCondition(sql, "org_id", config.getCurrentOrgId());
           } else if (config.getType() == DataPermissionType.ORG_AND_CHILDREN) {
               // WHERE org_id IN (SELECT id FROM org WHERE path LIKE '{currentOrgPath}%')
               addCondition(sql, "org_id", getOrgAndChildrenIds());
           } else if (config.getType() == DataPermissionType.SELF_ONLY) {
               // WHERE creator_id = {currentUserId}
               addCondition(sql, "creator_id", config.getCurrentUserId());
           }
           // ALL 类型不添加条件
       }
   }
   ```

【三、组织架构管理】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   - 组织变更历史：记录组织调整记录（影响的人员、站点、权限）。

2. 岗位管理（Position）：
   - 岗位定义：运维工程师、站长、区域经理、客服专员等。
   - 岗位关联角色：创建岗位时自动关联默认角色。

3. 用户管理：
   - 用户信息：账号、姓名、手机、邮箱、所属组织、岗位、状态（启用/禁用）。
   - 用户角色分配：支持按岗位自动分配 + 手动追加角色。
   - 用户数据权限配置：每个用户可单独配置数据权限范围。
   - 用户状态管理：
     * 启用/禁用（禁用后无法登录，已有会话强制下线）。
     * 锁定（连续 5 次登录失败自动锁定 30 分钟）。
     * 密码过期（90 天强制更换）。
   - 批量导入用户：Excel 批量导入（校验格式、去重、分配默认角色）。

【四、会话安全与审计】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 会话管理：
   - 登录策略：
     * 强密码策略：最少 8 位，含大小写+数字+特殊字符。
     * 双因素认证：强制开启 TOTP（Google Authenticator）。
     * IP 白名单：仅允许公司内网 IP 或 VPN IP 登录。
     * 设备绑定：首次登录绑定设备指纹，异地登录需重新验证。
   - 会话控制：
     * 单点登录：同一账号最多 3 个活跃会话。
     * 超时自动登出：30 分钟无操作自动登出。
     * 强制下线：管理员可强制下线指定用户的会话。

2. 操作审计日志（不可篡改）：
   ```java
   @Aspect
   @Component
   public class AuditLogAspect {
       @Around("@annotation(auditLog)")
       public Object audit(ProceedingJoinPoint pjp, AuditLog auditLog) {
           // 记录操作前数据快照
           Object beforeData = getBeforeData(pjp);
           
           Object result = pjp.proceed();
           
           // 记录操作后数据快照
           Object afterData = getAfterData(pjp);
           
           // 写入审计日志（异步写入独立的审计数据库）
           auditLogService.save(AuditLog.builder()
               .userId(currentUser.getId())
               .userName(currentUser.getName())
               .operation(auditLog.operation())
               .module(auditLog.module())
               .targetType(auditLog.targetType())
               .targetId(auditLog.targetId())
               .beforeData(JSON.toJSONString(beforeData))
               .afterData(JSON.toJSONString(afterData))
               .ip(getClientIp())
               .userAgent(getUserAgent())
               .result(result != null ? "SUCCESS" : "FAILED")
               .build());
           
           return result;
       }
   }
   ```

3. 敏感操作二次验证：
   - 删除站点、修改电价、批量操作、导出数据等敏感操作需输入操作密码。
   - 批量删除操作需主管审批。

要求：
- 权限数据缓存在 Redis，变更时主动失效（Pub/Sub 通知所有节点）。
- 权限校验必须覆盖所有 API（网关层 + 服务层双重校验）。
- 审计日志存储到独立的审计数据库（只追加，不可修改/删除），保留 3 年。
- 支持权限克隆：复制某个用户的权限给另一个用户。
- 支持权限矩阵视图：以表格形式展示各角色的权限配置（✅/❌）。
- 所有权限变更必须有审批流程（超级管理员除外）。
```

## 提示词 2：站点全生命周期管理与智能选址

> **定位**：管理充电站从选址、建设、上线到运营的全生命周期，并提供基于大数据的智能选址建议。

```text
请生成充电平台"后台管理后端"的"站点全生命周期管理与智能选址"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + PostgreSQL (PostGIS) + Redis + Elasticsearch + MinIO。
设计目标：支撑 10000+ 站点管理，支持 GIS 空间查询，提供智能选址评分。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、站点基础信息管理】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 站点数据模型（Station）：
   ```java
   public class Station {
       Long id;
       String stationNo;           // 站点编号 (ST-BJ-001)
       String tenantId;            // 租户ID
       
       // 基础信息

       String type;                // 类型：PUBLIC(公共)/PRIVATE(专用)/DESTINATION(目的地)

       Instant createdAt;
       Instant updatedAt;
   }

2. 站点状态机：
   ```
   PLANNING（规划中）
     → BUILDING（建设中）→ [验收通过] → ONLINE（运营中）
     → BUILDING → [验收不通过] → PLANNING
   ONLINE
     → MAINTENANCE（维护中）→ [维护完成] → ONLINE
     → OFFLINE（已下线）→ [重新上线] → ONLINE
   OFFLINE
     → DEMOLISHED（已拆除）
   ```

3. 站点 CRUD API：
   - POST /api/v1/admin/stations：创建站点（含地理位置、设施配置、电力信息）。
   - GET /api/v1/admin/stations：分页查询（支持按城市、类型、状态、运营商筛选）。
   - GET /api/v1/admin/stations/{id}：站点详情（含设备列表、运营数据、工单历史）。
   - PUT /api/v1/admin/stations/{id}：编辑站点。
   - DELETE /api/v1/admin/stations/{id}：删除站点（需审批，有运营数据的站点禁止删除）。
   - POST /api/v1/admin/stations/{id}/status：变更状态（上线/下线/维护）。

【二、GIS 空间查询与地图功能】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 空间查询（PostGIS）：
   - 按范围查询站点（圆形区域）：
     ```sql
     SELECT * FROM stations 
     WHERE ST_DWithin(
       location::geography, 
       ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)::geography, 
       :radius
     )
     AND tenant_id = :tenantId
     AND status = 'ONLINE';
     ```
   - 按多边形区域查询（电子围栏内站点）：
     ```sql
     SELECT * FROM stations 
     WHERE ST_Contains(:polygon::geometry, location::geometry)
     AND tenant_id = :tenantId;
     ```
   - 查询站点覆盖热力图（统计网格内站点密度）：
     ```sql
     SELECT 
       ST_AsGeoJSON(ST_SnapToGrid(location, 0.01)) as grid,
       COUNT(*) as density
     FROM stations
     WHERE city = :city
     GROUP BY grid;
     ```

2. 站点地图功能：
   - 地图模式查看站点分布（聚合/散点）。
   - 点击站点显示摘要信息（设备数、利用率、营收）。
   - 框选区域内的站点批量操作。
   - 站点密度热力图。
   - 竞品站点标注（手动添加竞品站点位置）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、智能选址评估系统】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 选址评分模型（多维度加权评分）：

   维度1 - 需求密度（权重 30%）：
   - 周边 3km 范围内新能源汽车保有量（来自车管所数据/估算）。
   - 周边 3km 范围内充电桩搜索热度（来自百度/高德指数）。
   - 周边住宅小区数量 × 平均车位比。
   - 周边商业/办公区域密度。

   维度2 - 竞争分析（权重 20%）：
   - 周边 1km 范围内竞品充电站数量。
   - 竞品站点总枪数/利用率/价格。
   - 市场饱和度 = 周边充电需求 / 周边充电供给。

   维度3 - 电力条件（权重 20%）：
   - 附近变压器容量余量（来自电网数据/人工勘察）。
   - 电力接入成本估算（距离最近接入点距离）。
   - 电价水平（峰谷电价差）。

   维度4 - 场地条件（权重 15%）：
   - 场地租金水平。
   - 停车位数量。
   - 场地可达性（主干道/支路、出入口便利度）。
   - 配套设施完善度。

   维度5 - 政策与补贴（权重 15%）：
   - 当地充电设施建设补贴额度。
   - 地方政策支持力度。
   - 土地性质合规性。

2. 选址报告生成：
   - 输入：选址坐标 + 半径。
   - 输出：PDF 格式的选址评估报告（含评分、各维度分析、竞品分布图、电力评估、投资建议）。

     * 运营商（主要收益方）。
     * 场地方/物业（场地租金或分成）。
     * 投资方（投资回报分成）。
     * 渠道方（引流分成，如高德、百度地图）。
   - 分账规则配置：
     * 按站点配置分账比例（如：平台 20% + 运营商 60% + 物业 15% + 渠道 5%）。
     * 电费和服务费可分别设置不同分账比例。
     * 分账比例之和必须 = 100%。
   - 分账计算示例：
     ```
     订单金额 = 电费 ¥30 + 服务费 ¥10 = ¥40
     电费分账：运营商 80% = ¥24，平台 20% = ¥6
     服务费分账：运营商 60% = ¥6，物业 30% = ¥3，平台 10% = ¥1
     平台总分账 = ¥6 + ¥1 = ¥7
     运营商总分账 = ¥24 + ¥6 = ¥30
     物业总分账 = ¥3
     ```

   - GET /api/v1/admin/settlements：分页查询（支持按站点、时间、状态筛选）。
   - GET /api/v1/admin/settlements/{id}：结算单详情。
   - POST /api/v1/admin/settlements/export：导出结算单 Excel。
   - POST /api/v1/admin/settlements/batch-settle：批量结算（手动触发）。

1. 对账任务（T+1 日对账）：
   - 定时任务（每日凌晨 2:00）：
     a. 下载各支付渠道前一天的对账文件：
        * 微信支付：调用微信对账文件下载 API，获取 CSV 文件。
        * 支付宝：调用支付宝对账文件下载 API。
        * 银联：调用银联对账文件下载 API。
     b. 解析对账文件，逐笔与本地订单匹配：
        * 匹配规则：商户订单号 + 金额。
     c. 差异分类：
        * 长款（渠道有，平台无）：可能是支付成功但平台未收到回调。
        * 短款（平台有，渠道无）：可能是支付超时或掉单。

     e. 差异处理：
