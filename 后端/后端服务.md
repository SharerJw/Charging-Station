# 🏗️ 充电平台 - 后端服务（四端统一支撑）深度细化提示词

针对**后端服务架构**，我将其拆分为 **7个核心模块** 进行深度细化。这些提示词涵盖了从微服务基座、百万级设备并发接入、高并发交易、极致性能、金融级稳定性、零信任安全到全链路可观测性的完整后端能力矩阵，**性能、稳定性、安全性全部拉满**。

## 提示词 1：云原生微服务基座与 API 智能网关

> **定位**：整个后端系统的骨架。包含微服务划分、服务治理、API 网关、多端 BFF 层、多租户路由、灰度发布。

```text
请生成充电平台后端的"云原生微服务基座与 API 智能网关"全套架构代码与配置。
技术栈：Java 21 (虚拟线程) + Spring Boot 3.3 + Spring Cloud Alibaba 2023 + Nacos 2.3 + Sentinel 1.8 + Apache APISIX (或 Spring Cloud Gateway) + Kubernetes + Helm。
构建工具：Gradle 8 + Kotlin DSL，多模块项目。
代码规范：阿里巴巴 Java 开发手册 + Google Java Style。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、微服务领域划分（DDD 领域驱动设计）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

采用 DDD 战略设计，将系统划分为以下限界上下文（微服务）：

1. gateway-service（API 网关服务）
2. identity-service（用户身份与认证服务）：用户注册/登录/OAuth2/JWT/权限。
3. station-service（站点管理服务）：站点 CRUD、地理围栏、配套设施。
4. device-service（设备管理服务）：设备台账、物模型、参数配置、OTA。
5. ocpp-service（OCPP 通信引擎服务）：WebSocket 长连接、OCPP 协议解析、指令收发。
6. charging-service（充电业务服务）：充电订单状态机、启停控制、实时计量。
7. order-service（订单与交易服务）：订单结算、退款、异常处理。
8. payment-service（支付与清结算服务）：多渠道支付、对账、分账、发票。
9. pricing-service（电价策略服务）：分时电价、动态定价、电价审批。
10. marketing-service（营销服务）：优惠券、活动、充值、积分、分销。
11. maintenance-service（运维服务）：工单、巡检、备件、告警。
12. analytics-service（数据分析服务）：报表、BI、用户画像、预测。
13. notification-service（消息通知服务）：短信、推送、邮件、微信模板消息。
14. admin-service（后台管理服务）：系统配置、角色权限、审计日志。
15. simulator-service（模拟器管理服务）：虚拟设备管理、场景编排、压测调度。

每个微服务内部采用经典 DDD 分层：
├── interfaces/         # 用户接口层（Controller, DTO, Assembler）
├── application/        # 应用服务层（ApplicationService, Command/Query, Event）
├── domain/             # 领域层（Entity, ValueObject, AggregateRoot, DomainService, Repository接口）
├── infrastructure/     # 基础设施层（Repository实现, 外部API适配器, 消息队列, 缓存）
└── shared/             # 共享内核（通用枚举、异常、工具类）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、API 网关设计（gateway-service）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

采用 Apache APISIX 作为流量入口（或 Spring Cloud Gateway 作为备选），承担以下职责：

1. 统一路由与多端适配：
   - 路由规则（按端区分）：
     * /api/v1/app/**    → 用户小程序端（限流严格、需用户Token）
     * /api/v1/admin/**  → 后台管理Web端（需管理员Token + RBAC校验）
     * /api/v1/ops/**    → 运维App端（需运维Token + 设备权限校验）
     * /api/v1/sim/**    → 模拟器Web端（需API Key认证）
     * /ws/ocpp/**       → OCPP WebSocket 长连接（直连 ocpp-service）
   - BFF（Backend For Frontend）聚合层：
     * 小程序首页需要聚合站点+电价+空闲状态 → 在 gateway 层做请求编排（Parallel Call + Merge）。
     * 使用 APISIX 的 Serverless 插件或独立的 BFF 微服务实现。

2. 统一鉴权过滤器（Auth Filter）：
   - JWT Token 校验（RS256 非对称签名，公钥缓存在网关内存）。
   - Token 续期：Access Token 有效期 2h，Refresh Token 有效期 7d，滑动窗口续期。
   - 多端 Token 隔离：不同类型端使用不同的 JWT Claim（如 "client_type": "app" | "admin" | "ops"）。
   - API Key 认证（模拟器端/第三方对接）：HMAC-SHA256 签名校验。
   - 设备证书认证（OCPP TLS 连接）：mTLS 双向证书校验。

3. 流量治理：
   - 限流策略（Sentinel 集成）：
     * 全局限流：10,000 QPS。
     * 单用户限流：100 QPS（防刷）。
     * 单 IP 限流：200 QPS。
     * 接口级限流：充电启动接口 500 QPS，查询接口 5,000 QPS。
   - 熔断降级：下游服务响应 > 3s 或错误率 > 50% 时自动熔断，返回降级数据。
   - 灰度发布：基于请求 Header 中的用户标签/设备ID，按比例或按规则路由到 Canary 版本。
   - 请求重写：URL 重写、Header 注入（注入 X-User-Id, X-Tenant-Id, X-Trace-Id）。

4. 安全防护：
   - CORS 白名单配置（仅允许自有域名）。
   - 请求体大小限制（10MB）。
   - SQL 注入/XSS 过滤（WAF 插件）。
   - Bot 防护：User-Agent 黑名单 + 行为频率检测。
   - 敏感接口防重放：Nonce + Timestamp 校验（5分钟窗口）。

5. 可观测性：
   - 每个请求生成全局唯一 TraceId（W3C Trace Context 标准），注入 Header 透传全链路。
   - 请求/响应日志异步写入 Kafka → ELK。
   - Prometheus 指标暴露（QPS、延迟分位数、错误率、状态码分布）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、服务注册与配置中心（Nacos）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 服务注册：
   - 所有微服务启动时自动注册到 Nacos，元数据包含：版本号、权重、区域、灰度标签。
   - 健康检查：Nacos 心跳 + Spring Boot Actuator /health 端点双重检测。
   - 服务发现：使用 Spring Cloud LoadBalancer（客户端负载均衡），策略为加权轮询。

2. 配置管理：
   - 配置分层：全局共享配置 → 环境级配置(dev/test/staging/prod) → 服务级配置 → 灰度配置。
   - 敏感配置加密：数据库密码、API Key 使用 Nacos 配置加密功能（AES-256）。
   - 动态刷新：使用 @RefreshScope 或 @ConfigurationProperties 实现配置热更新（无需重启），关键配置（如电价规则、限流阈值）变更秒级生效。
   - 配置变更审计：所有配置变更记录操作人、变更前后对比、时间戳。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【四、多租户架构】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

支持 SaaS 多租户模式（一个平台服务多个充电运营商）：

1. 租户识别策略：
   - HTTP 请求：从 Header X-Tenant-Id 或 JWT Token 中提取。
   - WebSocket 连接：从 OCPP 连接 URL 路径中提取（如 /ws/ocpp/{tenantId}/{chargePointId}）。

2. 数据隔离策略（可配置）：
   - 共享数据库 + 租户字段隔离（默认，成本低）：所有表带 tenant_id 字段，MyBatis-Plus 自动拼接租户条件。
   - 独立 Schema 隔离（高级租户）：每个租户独立数据库 Schema，动态数据源路由。
   - 独立数据库隔离（VIP 租户）：完全独立数据库实例。

3. 租户上下文透传：
   - 使用 ThreadLocal（虚拟线程兼容的 ScopedValue）存储当前租户ID。
   - 异步/消息场景：租户ID 通过 MDC（日志）+ Message Header 透传。

要求：
- 所有微服务必须实现优雅启停（Graceful Shutdown）：收到 SIGTERM 后停止接收新请求，等待进行中的请求处理完毕（最长 30s），再从 Nacos 注销。
- 使用 Java 21 虚拟线程（Virtual Threads），所有 I/O 密集型操作使用虚拟线程，告别传统线程池调优。
- 所有微服务提供 OpenAPI 3.0 (Swagger) 文档，自动生成。
- 项目根目录提供 docker-compose.yml（一键启动所有中间件：Nacos/Redis/Kafka/MySQL/MinIO）。
- 提供 K8s Helm Chart，支持一键部署到 Kubernetes 集群。
```

## 提示词 2：百万级设备并发接入与 OCPP 通信引擎

> **定位**：整个充电平台最核心的技术难点。支撑百万级充电桩 WebSocket 长连接，高性能解析 OCPP 1.6J/2.0.1 协议，实现低延迟的指令收发与状态管理。

```text
请生成充电平台后端的"百万级设备并发接入与 OCPP 通信引擎"核心代码。
技术栈：Java 21 + Netty 4.1 (或 Go + gorilla/websocket，若选 Go 微服务) + Reactor (响应式编程) + Kafka + Redis + Caffeine。
设计目标：单节点支撑 10 万+ WebSocket 长连接，集群支撑 100 万+ 设备并发在线，消息处理延迟 < 50ms (P99)。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、WebSocket 接入层（Netty 高性能服务器）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Netty 服务器配置（核心参数调优）：
   - Boss Group：1 个 NioEventLoop（负责 Accept 连接）。
   - Worker Group：CPU核心数 × 2 个 NioEventLoop（负责 IO 读写）。
   - Business Group：使用 Java 21 虚拟线程 Executor（负责业务逻辑处理，不阻塞 Worker）。
   - Channel Pipeline 设计（从上到下）：
     * IdleStateHandler：读写空闲检测（心跳超时判断）。
     * HttpServerCodec：HTTP 编解码。
     * HttpObjectAggregator：HTTP 消息聚合。
     * WebSocketServerProtocolHandler：WebSocket 握手与帧处理。
     * OcppFrameDecoder：OCPP 消息帧解码（处理粘包/半包）。
     * OcppJsonCodec：JSON ↔ Java 对象序列化/反序列化（使用 Jackson + Afterburner 模块）。
     * RateLimitHandler：单连接限流（每秒最多 20 条消息，防设备风暴）。
     * OcppBusinessHandler：核心业务分发处理。
   - TCP 参数调优：
     * SO_REUSEADDR = true
     * TCP_NODELAY = true
     * SO_KEEPALIVE = true
     * SO_RCVBUF = 64KB / SO_SNDBUF = 64KB
     * writeBufferWaterMark = (32KB, 64KB)（防内存溢出）

2. 连接管理（ConnectionManager）：
   - 数据结构：ConcurrentHashMap<String, Channel>（deviceId → Channel 映射）。
   - 连接建立流程：
     * WebSocket 握手 → 从 URL 提取 deviceId + tenantId。
     * 设备身份认证（HTTP Basic Auth / Token / TLS 证书）。
     * 查询设备信息（Redis 缓存优先 → DB 降级）。
     * 设备状态检查（是否已在线？如已在线，踢掉旧连接并记录日志）。
     * 注册连接到 ConnectionManager + 更新 Redis 设备在线状态（带 TTL 过期）。
     * 发送欢迎消息 / 触发 BootNotification 处理。
   - 连接断开处理：
     * 正常关闭：收到 CloseFrame → 清理资源 → 更新离线状态。
     * 异常断开：触发 channelInactive → 延迟 10s 检查是否重连 → 未重连则标记离线 + 触发告警。
   - 连接迁移（集群场景）：
     * 使用一致性哈希将设备分配到网关节点。
     * 节点扩缩容时，通过 Redis Pub/Sub 通知设备重连到新节点。

3. 心跳与保活机制：
   - 双重心跳：
     * WebSocket 协议层 Ping/Pong（Netty 自动处理，间隔 30s）。
     * OCPP 业务层 Heartbeat（设备主动上报，间隔由后端通过 HeartbeatInterval 配置控制，默认 60s）。
   - 心跳超时判定：连续 3 次未收到任何消息（Ping/Pong 或业务消息）→ 判定离线 → 关闭连接。
   - 心跳响应优化：收到 Heartbeat 后直接返回当前服务器时间戳，无需查库。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、OCPP 协议解析引擎】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. OCPP 消息类型处理：
   - OCPP-J (JSON) 消息格式：[MessageTypeId, UniqueId, Action, Payload]
   - MessageTypeId 处理：
     * Type 2 (Call)：设备→后端 请求。后端处理后必须回复 Type 3 或 Type 4。
     * Type 3 (CallResult)：后端→设备 请求的成功响应（设备回复后端的指令）。
     * Type 4 (CallError)：请求的错误响应。

2. 上行消息处理器（设备→后端，按 Action 分类）：
   
   【注册与状态类】
   - BootNotificationHandler：
     * 校验设备信息（ChargePointVendor, ChargePointModel, FirmwareVersion 等）。
     * 判断是否允许注册（设备是否在白名单/是否已停用）。
     * 返回：Accepted / Pending / Rejected + HeartbeatInterval。
     * Accepted 后触发：初始化设备配置、同步参数模板、推送到 Kafka "device.registered" Topic。
   
   - StatusNotificationHandler：
     * 解析 ConnectorId + Status（Available/Occupied/Faulted/Unavailable 等）+ ErrorCode。
     * 更新 Redis 中设备/枪的实时状态（Hash 结构，TTL 5 分钟）。
     * 状态变更时推送到 Kafka "device.status.changed" Topic（供告警服务、前端实时消费）。
     * 如果状态为 Faulted，解析 ErrorCode 和 Info，触发告警创建流程。
   
   - HeartbeatHandler：
     * 更新设备最后心跳时间（Redis，O(1) 操作）。
     * 返回服务器当前时间（currentTime）。
   
   【充电业务类】
   - StartTransactionHandler：
     * 校验 IdTag 有效性（调用 charging-service 验证）。
     * 记录 transactionId（后端生成全局唯一雪花ID）+ connectorId + meterStart + timestamp。
     * 创建充电订单记录（状态=充电中）。
     * 返回：transactionId + idTagInfo（Accepted/Blocked/Expired/Invalid）。
     * 推送到 Kafka "charging.started" Topic。
   
   - StopTransactionHandler：
     * 校验 transactionId 匹配。
     * 记录 meterStop + timestamp + reason（停止原因）。
     * 解析 transactionData（分段计量数据，用于精确计费）。
     * 更新订单状态为"待结算"。
     * 推送到 Kafka "charging.stopped" Topic → 触发结算流程。
     * 返回：idTagInfo。
   
   - MeterValuesHandler：
     * 解析采样数据（Energy.Active.Import.Register, Power.Active.Import, Current, Voltage, SoC 等）。
     * 写入时序数据库（TDengine / InfluxDB / TimescaleDB），按 deviceId + connectorId + timestamp 组织。
     * 实时功率/SOC 推送到 Redis（供前端实时查询）和 Kafka（供大屏/监控消费）。
     * 采样频率控制：通过 MeterValueSampleInterval 配置项控制（默认 60s）。
   
   【远程指令响应类】
   - 处理后端下发指令的 Response（Type 3/4）：
     * 匹配 pending request（通过 UniqueId 关联）。
     * 使用 CompletableFuture.complete() 唤醒等待线程。
     * 超时处理：每个下发指令设置超时时间（默认 30s），超时后 CompletableFuture.completeExceptionally(TimeoutException)。

3. 下行指令下发器（后端→设备）：
   
   - 指令类型：
     * RemoteStartTransaction：远程启动充电。
     * RemoteStopTransaction：远程停止充电。
     * Reset (Hard/Soft)：远程重启。
     * UnlockConnector：远程解锁充电枪。
     * ChangeConfiguration：修改设备配置参数。
     * GetConfiguration：查询设备配置。
     * UpdateFirmware：固件升级。
     * ChangeAvailability：设置设备可用状态。
     * TriggerMessage：触发设备上报特定消息。
     * ReserveNow / CancelReservation：预约/取消预约。
   
   - 指令下发流程：
     * 1. 生成全局唯一 UniqueId（UUID）。
     * 2. 构建 OCPP Call 消息 [2, uniqueId, action, payload]。
     * 3. 注册到 PendingRequestManager（ConcurrentHashMap<uniqueId, CompletableFuture>）。
     * 4. 通过 Channel 写入并 flush 消息。
     * 5. 异步等待 CompletableFuture（带超时）。
     * 6. 返回结果给调用方（成功/失败/超时）。
   
   - 指令队列与流控：
     * 每个设备维护一个指令队列（LinkedBlockingQueue，容量 100）。
     * 同一设备同时只处理 1 条指令（串行化，避免设备混乱）。
     * 队列满时拒绝新指令并返回"设备忙"。
     * 重要指令（如 RemoteStop）可插队（Priority Queue）。
   
   - 离线指令缓存：
     * 设备离线时，指令存入 Redis List（待下发队列）。
     * 设备重连后，自动从队列中取出依次下发。
     * 过期指令自动清理（TTL 24h）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、集群架构与消息路由】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 多节点连接管理：
   - 设备连接信息存储在 Redis Cluster 中：
     * Key: "device:conn:{deviceId}"
     * Value: { nodeId, ip, port, connectedAt, lastHeartbeat }
     * TTL: 5 分钟（心跳续期）。
   - 跨节点指令路由：
     * 当 admin-service 要下发指令给 device-001 时：
       - 查 Redis 获取 device-001 连接的 ocpp-service 节点ID。
       - 通过 Kafka 发送指令消息到特定 Topic Partition（以 nodeId 为 Key）。
       - 目标 ocpp-service 节点消费消息后，通过本地 Channel 下发给设备。

2. 消息总线（Kafka Topic 设计）：
   - ocpp.upstream.raw     # 原始上行报文（用于审计/分析）
   - ocpp.downstream.raw   # 原始下行报文
   - device.status.changed # 设备状态变更事件
   - charging.started      # 充电开始事件
   - charging.stopped      # 充电结束事件
   - meter.values          # 计量数据事件
   - device.alert          # 设备告警事件
   - command.request       # 指令下发请求（跨节点路由）
   - command.response      # 指令执行结果

3. 性能优化关键措施：
   - 零拷贝：Netty 使用 FileRegion 和 CompositeByteBuf 减少内存拷贝。
   - 对象池：OCPP 消息对象使用 Recycler 对象池，减少 GC 压力。
   - 批量写入：MeterValues 数据积攒 100ms 后批量写入时序数据库。
   - 背压控制：当下游（Kafka/DB）处理不过来时，对设备消息进行背压（减缓读取速度）。
   - 内存管理：限制单个 Channel 的写缓冲区大小，防止慢设备拖垮整个节点。

要求：
- 所有 OCPP 消息处理必须幂等（重复消息不产生副作用）。
- 协议解析使用严格 JSON Schema 校验，不合规消息直接丢弃并记录告警。
- 提供 OCPP 模拟器对接测试套件（自动化测试 100+ 场景）。
- Netty 服务器支持平滑重启（新连接导流到其他节点，旧连接等待处理完毕）。
- 所有网络通信支持 TLS 1.3 加密（OCPP 2.0.1 强制要求）。
- 连接数/消息吞吐量/延迟等核心指标暴露 Prometheus metrics。

## 提示词 3：高并发交易与清结算中心

> **定位**：处理充电订单的全生命周期（创建→计量→结算→支付→退款→对账→分账），保证资金流转的绝对正确性和高并发处理能力。

```text
请生成充电平台后端的"高并发交易与清结算中心"全套代码。
技术栈：Java 21 + Spring Boot 3.3 + MyBatis-Plus + Seata (分布式事务) + RocketMQ (事务消息) + Redis + MySQL 8.0 (InnoDB) + ShardingSphere。
设计目标：支撑日均 500 万笔充电订单，结算延迟 < 3 秒，资金零差错。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、订单状态机（核心领域模型）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

充电订单使用有限状态机（FSM）严格管控状态流转，禁止非法跳转。

状态定义（OrderStatus 枚举）：
  CREATED          → 订单创建（设备上报 StartTransaction）
  CHARGING         → 充电中
  STOPPING         → 停止中（下发 RemoteStop 或设备上报 StopTransaction）
  STOPPED          → 已停止（充电物理结束，等待结算）
  SETTLING         → 结算中（计算费用）
  SETTLED          → 已结算（费用计算完成，等待支付）
  PAYING           → 支付中
  PAID             → 已支付（交易完成）
  REFUNDING        → 退款中
  REFUNDED         → 已退款
  ABNORMAL         → 异常（需人工介入）
  CANCELLED        → 已取消

状态流转规则（StateMachine 配置）：
  CREATED → CHARGING      [触发：StartTransaction 确认]
  CHARGING → STOPPING     [触发：用户点击停止 / 充满 / 远程停止 / 设备异常]
  STOPPING → STOPPED      [触发：StopTransaction 确认]
  STOPPED → SETTLING      [触发：自动结算任务（StopTransaction 后立即触发）]
  SETTLING → SETTLED      [触发：费用计算完成]
  SETTLED → PAYING        [触发：发起支付扣款]
  PAYING → PAID           [触发：支付成功回调]
  PAID → REFUNDING        [触发：退款申请]
  REFUNDING → REFUNDED    [触发：退款成功回调]
  任意状态 → ABNORMAL     [触发：异常检测（超时/数据不一致）]

实现方式：
  - 使用 Spring StateMachine 或自研轻量级状态机。
  - 每次状态变更：
    1. 校验当前状态是否允许转换到目标状态。
    2. 执行状态变更前置动作（Guard）。
    3. 更新数据库状态（使用乐观锁：UPDATE orders SET status = ?, version = version + 1 WHERE id = ? AND status = ? AND version = ?）。
    4. 执行状态变更后置动作（Action）。
    5. 记录状态变更日志（audit_order_status 表）。
    6. 发布领域事件（OrderStatusChangedEvent）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、计费引擎（核心算法）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 分段计费算法（支持跨时段充电）：
   
   场景：用户在 14:30 开始充电，17:45 结束，跨越了平段(14:00-17:00)和峰段(17:00-21:00)。
   
   算法步骤：
   a. 获取该站点的分时电价配置（时段列表 + 各时段电费/服务费单价）。
   b. 获取充电过程中的分段电量数据（来自 MeterValues 或 StopTransaction.transactionData）。
      - 如果有精确的分段电量数据：直接按时段匹配计算。
      - 如果只有总电量：按时间比例 + 功率曲线插值算法估算各时段电量。
   c. 各时段费用计算：
      - 时段电费 = 该时段充电量(kWh) × 该时段电费单价
      - 时段服务费 = 该时段充电量(kWh) × 该时段服务费单价
   d. 汇总：
      - 总电费 = Σ 各时段电费
      - 总服务费 = Σ 各时段服务费
      - 原始总费用 = 总电费 + 总服务费
   
   精度保证：
   - 所有金额使用 BigDecimal，RoundingMode.HALF_UP，保留 2 位小数。
   - 数据库金额字段使用 DECIMAL(10,2)。
   - 内部计算使用"分"（Long 类型）避免浮点误差，仅在展示时转换为"元"。

2. 优惠叠加计算（按优先级顺序）：
   a. 原始费用 = 电费 + 服务费。
   b. 会员折扣：原始费用 × 会员折扣率（如金卡 0.95）。
   c. 优惠券抵扣：
      - 满减券：满足门槛后直接减免。
      - 折扣券：折后金额 = 费用 × 折扣率（设最高优惠上限）。
      - 电量券：直接减免对应电量的费用。
      - 多券不可叠加（取最优的一张），或按规则叠加（配置化）。
   d. 积分抵扣：积分 ÷ 兑换比例（如 100积分 = 1元）。
   e. 最终费用 = max(0.01, 折后费用 - 各抵扣之和)（最低 0.01 元，防止零元单）。

3. 计费结果存储：
   - order_billing 表：订单ID、总电费、总服务费、各时段明细(JSON)、优惠明细(JSON)、最终金额。
   - 计费快照：将计算时使用的电价版本、优惠规则版本一并存储（确保可复现、可审计）。

【三、支付与资金流转】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 支付渠道抽象（策略模式）：
   - PaymentChannel 接口：createOrder / queryStatus / refund / close。
   - 实现类：WechatPayChannel / AlipayChannel / BalancePayChannel / EnterprisePayChannel。
   - 支付路由：根据用户选择的支付方式 + 金额 + 渠道可用性自动路由。

2. 微信支付分"先充后付"流程：
   a. 充电开始前：调用微信支付分 API 创建"服务订单"（预授权）。
   b. 充电中：实时更新预估费用（可选，用于前端展示）。
   c. 充电结束后：
      - 调用结算引擎计算最终费用。
      - 调用微信支付分 API "完结订单" → 微信自动从用户账户扣款。
      - 接收微信支付回调 → 更新订单状态为 PAID。
   d. 扣款失败处理：
      - 重试扣款（指数退避，最多 3 次）。

3. 余额支付：
   - 使用 Redis + Lua 脚本保证扣减原子性：
     ```
     -- KEYS[1] = user:balance:{userId}
     -- ARGV[1] = amount (分)
     local balance = redis.call('GET', KEYS[1])
     if tonumber(balance) >= tonumber(ARGV[1]) then
         redis.call('DECRBY', KEYS[1], ARGV[1])
         return 1  -- 成功
     else
         return 0  -- 余额不足
     end
     ```
   - 扣减成功后写入 balance_flow 流水表（金额、类型、关联订单ID、余额快照）。
   - 使用 RocketMQ 事务消息保证"扣余额 + 更新订单状态"的最终一致性。

4. 退款处理：
   - 退款申请 → 审批（根据金额走不同审批流）→ 审批通过 → 调用原支付渠道退款 API。
   - 退款幂等：每笔退款生成唯一 refund_no，渠道侧做幂等校验。
   - 部分退款：支持退还部分金额（如多扣的费用）。
   - 退款资金回补：余额支付的退款直接回补余额 + 记录流水。

【四、对账与分账系统】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 提示词 4：极致性能优化与缓存/消息中间件实战

> **定位**：针对四端不同场景的极致性能优化方案，包括多级缓存架构、消息队列削峰填谷、数据库读写分离与分库分表、异步化处理。

```text
请生成充电平台后端的"极致性能优化"全套方案代码。
技术栈：Caffeine (L1 本地缓存) + Redis Cluster (L2 分布式缓存) + Kafka/RocketMQ + MySQL 8.0 (读写分离 + 分库分表) + TDengine (时序数据库) + Elasticsearch。
设计目标：核心接口 P99 延迟 < 100ms，支撑 10 万 QPS 查询，1 万 TPS 写入。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、多级缓存架构（L1 + L2 + L3）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 缓存分层设计：
   - L1 本地缓存（Caffeine）：
     * 存储：电价配置、站点基础信息、设备物模型模板、数据字典。
     * 容量：每个节点最多 10,000 个 Key。
     * 过期策略：写入后 5 分钟过期 + 最大空闲 10 分钟过期。
     * 命中率目标：> 95%。
   
   - L2 分布式缓存（Redis Cluster，6 节点 3 主 3 从）：
     * 存储：设备实时状态、用户会话、订单快照、站点空闲枪数、排行榜。
     * 过期策略：根据数据类型设置不同 TTL（设备状态 5min，用户会话 2h，订单快照 24h）。
     * 容量规划：64GB 集群，预留 40% 内存给 AOF/RDB。
   
   - L3 持久化存储（MySQL / TDengine / ES）：
     * 缓存未命中时的最终数据来源。

2. 缓存一致性保障（核心难点）：
   
   策略一：Cache-Aside + 延迟双删（适用于站点信息、电价配置等低频变更数据）
   ```java
   // 更新操作
   public void updateStation(StationDTO dto) {
       // 1. 先更新数据库
       stationMapper.updateById(dto);
       // 2. 删除 Redis 缓存
       redisTemplate.delete("station:" + dto.getId());
       // 3. 异步延迟 500ms 再删一次（防止并发读导致旧值回填）
       delayedExecutor.schedule(() -> {
           redisTemplate.delete("station:" + dto.getId());
           // 4. 同时清除所有节点的 L1 本地缓存
           kafkaTemplate.send("cache.invalidate", "station:" + dto.getId());
       }, 500, TimeUnit.MILLISECONDS);
   }
   ```
   
   策略二：Redis Pub/Sub 广播失效（适用于 L1 本地缓存同步）
   - 当某个节点更新了数据，发布失效消息到 Redis Channel。
   - 所有节点订阅该 Channel，收到消息后清除本地 Caffeine 对应 Key。
   
   策略三：Binlog 监听 + 异步刷新（适用于订单状态等高频变更数据）
   - 使用 Canal/Debezium 监听 MySQL Binlog。
   - 数据变更时自动推送 Kafka 消息 → 缓存刷新消费者更新 Redis。
   
   策略四：版本号机制（适用于电价配置）
   - 每次电价变更，版本号 +1。
   - 缓存中存储 { data, version }，读取时对比版本号，不一致则刷新。

3. 缓存穿透/击穿/雪崩防护：
   - 穿透防护：
     * 布隆过滤器（Redis Bloom Filter）拦截不存在的设备ID/用户ID。
     * 空值缓存：查询为空时缓存 NULL 值，TTL 5 分钟。
   - 击穿防护：
     * 互斥锁（Redis SETNX）：缓存失效时，只允许一个线程回源查询，其他线程等待。
     * 逻辑过期：缓存不设物理 TTL，value 中包含过期时间戳，过期后异步刷新，旧值继续返回。
   - 雪崩防护：
     * TTL 随机化：基础 TTL + random(0, 300) 秒，避免大量 Key 同时过期。
     * 多级降级：Redis 不可用时降级到 L1 本地缓存 → 再降级到 DB 限流查询。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、消息队列实战（削峰填谷 + 异步解耦）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Kafka 集群配置（3 Broker，3 副本）：
   
   Topic 规划与分区策略：
   | Topic                    | Partitions | 消费者组              | 说明                    |
   |--------------------------|-----------|----------------------|-------------------------|
   | ocpp.upstream.raw        | 32        | ocpp-raw-consumer    | 原始上行报文（高吞吐）    |
   | device.status.changed    | 16        | status-consumer      | 设备状态变更              |
   | charging.started         | 16        | charging-consumer    | 充电开始事件              |
   | charging.stopped         | 16        | settlement-consumer  | 充电结束→触发结算          |
   | meter.values             | 32        | meter-consumer       | 计量数据（最高频）         |
   | payment.callback         | 8         | payment-consumer     | 支付回调                  |
   | notification.send        | 8         | notification-consumer| 通知发送                  |
   | analytics.event          | 16        | analytics-consumer   | 埋点事件                  |

2. 生产者优化：
   - 批量发送：linger.ms = 10ms，batch.size = 64KB。
   - 压缩：使用 LZ4 压缩（压缩比高，CPU 开销低）。
   - ACK 策略：核心事件（充电开始/结束/支付）使用 acks=all，非核心事件（埋点/日志）使用 acks=1。
   - 分区 Key：使用 deviceId 作为 Key，保证同一设备的消息有序。

3. 消费者优化：
   - 批量消费：max.poll.records = 500，批量写入 DB/ES。
   - 并行处理：使用虚拟线程池并行处理消费到的消息。
   - 死信队列（DLQ）：消费失败 3 次后进入死信 Topic，人工介入处理。
   - 消费延迟监控：Burrow 监控 Consumer Lag，延迟 > 10000 条时告警。

4. 典型异步化场景：
   - 充电结束后：
     * 同步：更新订单状态为 STOPPED → 返回设备响应（< 50ms）。
     * 异步（Kafka）：触发结算 → 触发积分发放 → 触发通知推送 → 触发数据归档 → 触发用户画像更新。
   - 用户注册后：
     * 同步：创建用户记录 → 返回 Token。
     * 异步：发放新人礼包 → 发送欢迎短信 → 创建用户画像 → 记录注册来源。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、数据库性能优化】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 读写分离（ShardingSphere-JDBC）：
   - 1 主 2 从架构，主库写入，从库查询。
   - 强制走主库场景：支付回调更新订单后立即查询（使用 Hint 强制路由主库）。
   - 从库延迟监控：延迟 > 1s 时告警，> 5s 时自动切换查询到主库。

2. 分库分表策略：
   - charging_order 表：按 user_id 哈希分 16 表（单表控制在 500 万行以内）。
   - meter_values 数据：使用时序数据库 TDengine，按 device_id + connector_id 建超级表，自动按天分片。
   - 日志表（操作日志/通信日志）：按月分表 + 自动归档（超过 6 个月的数据迁移到冷存储）。

3. 索引优化：
   - 所有查询必须走索引（Explain 验证，禁止全表扫描）。
   - 复合索引遵循最左前缀原则。
   - 高频查询使用覆盖索引（避免回表）。
   - 慢查询监控：超过 200ms 的查询自动记录 + 告警。

4. 连接池优化（HikariCP）：
   - maximumPoolSize = CPU核心数 × 2 + 磁盘数（通常 20-30）。
   - connectionTimeout = 3000ms。
   - idleTimeout = 600000ms。
   - maxLifetime = 1800000ms。
   - 泄露检测：leakDetectionThreshold = 60000ms。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【四、接口性能优化技巧】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 并行调用（CompletableFuture + 虚拟线程）：
   ```java
   // 小程序首页数据聚合：并行调用站点服务+电价服务+设备状态服务
   var stations = CompletableFuture.supplyAsync(() -> stationService.getNearby(lat, lng), vtExecutor);
   var prices = CompletableFuture.supplyAsync(() -> pricingService.getPrices(stationIds), vtExecutor);
   var statuses = CompletableFuture.supplyAsync(() -> deviceService.getBatchStatus(deviceIds), vtExecutor);
   CompletableFuture.allOf(stations, prices, statuses).join();
   // 聚合结果返回
   ```

2. 响应压缩：Gzip 压缩（> 1KB 的 JSON 响应自动压缩）。

3. 接口缓存注解：
   ```java
   @Cacheable(value = "station", key = "#id", unless = "#result == null")
   @CacheTTL(seconds = 300, randomOffset = 60)  // 自定义注解，TTL 随机偏移防雪崩
   public StationDTO getStation(Long id) { ... }
   ```

4. 分页查询优化：
   - 禁止深分页（OFFSET > 10000）。
   - 使用游标分页（Cursor-based）：WHERE id > lastId ORDER BY id LIMIT 20。
   - 或使用 ES 做复杂条件搜索 + 分页。

要求：
- 所有核心接口必须有性能基准测试（JMeter/wrk），提供压测报告。
- 缓存命中率、消息队列延迟、数据库 QPS 等指标接入 Grafana 实时监控。
- 数据库变更必须通过 Flyway/Liquibase 管理（版本化迁移脚本）。
- 所有异步操作必须有超时控制和重试机制（Spring Retry + 指数退避）。
```

## 提示词 5：金融级高可用与分布式稳定性保障

> **定位**：确保系统 7×24 小时不间断运行，99.99% 可用性。涵盖多活容灾、限流熔断降级、分布式事务、数据一致性、故障演练。

```text

设计目标：系统可用性 99.99%（全年宕机 < 52 分钟），RPO = 0（数据零丢失），RTO < 5 分钟（故障恢复时间）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、多活容灾架构】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 同城双活（Active-Active）：
   - 两个可用区（AZ-A / AZ-B），相距 < 50km，光纤延迟 < 1ms。
   - 流量分配：50% / 50%（通过 DNS 加权轮询或 GSLB 全局负载均衡）。

     * Kafka：3 Broker 分布在不同 AZ，replication.factor = 3，min.insync.replicas = 2。

     * AZ-A 整体故障时，GSLB 自动将流量切到 AZ-B（DNS TTL = 30s）。

2. 异地灾备（Active-Standby）：
   - 异地数据中心（距离 > 500km），作为冷备。
   - 数据异步复制：MySQL Binlog → Canal → 异地 MySQL（延迟 < 5s）。
   - 定期灾备演练（每季度一次）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、限流、熔断与降级（Sentinel 实战）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 限流策略（多层防护）：

   层级1 - 网关层限流（APISIX/Spring Cloud Gateway）：
   - 全局限流：系统总 QPS 上限 50,000。
   - IP 限流：单 IP 200 QPS。
   - 用户限流：单用户 100 QPS。

   层级2 - 服务层限流（Sentinel）：
   - 接口级限流：
     * /api/v1/app/stations/nearby → 5,000 QPS
     * /api/v1/app/charging/start → 500 QPS
     * /api/v1/app/payment/create → 1,000 QPS
   - 自适应保护：系统 CPU 使用率 > 80% 或 Load > 5 时自动拒绝新请求。
   - 热点参数限流：对特定 stationId 或 deviceId 的查询限流（防止热点设备被打爆）。

   限流响应：
   - HTTP 429 Too Many Requests + Retry-After Header。
   - 统一响应体：{ "code": 429, "message": "当前访问人数过多，请稍后重试", "retryAfter": 5 }

2. 熔断策略：
   - 慢调用熔断：接口平均响应时间 > 3s 且请求数 > 10/min → 熔断 30s。
   - 异常比例熔断：异常比例 > 50% 且请求数 > 10/min → 熔断 30s。
   - 异常数熔断：异常数 > 5/min → 熔断 30s。
   - 熔断恢复：熔断 30s 后进入 Half-Open 状态，放行 1 个请求探测，成功则恢复，失败则继续熔断。

3. 降级策略（按优先级分级）：
   
   降级 Level 1（轻微降级，自动触发）：
   - 关闭非核心功能：积分发放延迟、评价推送延迟、充电报告生成延迟。
   - 降低数据实时性：站点空闲枪数从实时改为 1 分钟缓存。
   
   降级 Level 2（中度降级，自动/手动触发）：
   - 充电功率曲线停止记录（仅保留关键计量数据）。
   - 推荐算法降级为默认推荐（不调用 AI 模型）。
   - 站点图片/VR 不加载。
   
   降级 Level 3（严重降级，手动触发）：
   - 仅保留核心充电链路（扫码→充电→支付），暂停所有其他功能。
   - 注册/登录降级（仅允许已登录用户充电）。
   - 营销功能全部关闭。
   
   降级 Level 4（紧急降级，手动触发）：
   - 全站静态化（返回预生成的静态页面）。
   - 充电功能暂停，显示维护公告。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 事务策略选择（按场景）：

   场景A - 强一致性要求（支付扣款 + 订单状态更新）：
   - 方案：RocketMQ 事务消息（最终一致性）。
   ```java
   // 1. 发送半消息（Half Message）
   Message msg = new Message("payment_topic", paymentJson.getBytes());
   TransactionSendResult result = producer.sendMessageInTransaction(msg, null);
   
   // 2. 执行本地事务（扣减余额）
   // 3. 根据本地事务结果 Commit 或 Rollback 消息
   
   // 4. 下游服务消费消息（更新订单状态）
   // 5. 如果下游消费失败，MQ 会重试投递
   // 6. 如果生产者未 Commit/Rollback，MQ 会回调 checkLocalTransaction 检查本地事务状态
   ```
   
   场景B - 跨服务数据一致性（充电结束 → 结算 → 积分 → 通知）：
   - 方案：Saga 模式（编排式）。
   - 使用 Seata Saga 引擎或自研事件编排：
     * Step 1: 结算服务计算费用 → 成功 → 触发 Step 2。
     * Step 2: 订单服务更新状态 → 成功 → 触发 Step 3。
     * Step 3: 积分服务发放积分 → 失败 → 补偿 Step 2 → 补偿 Step 1。
   - 每个 Step 必须实现正向操作和补偿操作（幂等）。
   
   场景C - 高并发写入（MeterValues 时序数据）：
   - 方案：异步写入 + 内存聚合。
   - 消息先入 Kafka → 消费者批量写入 TDengine（每 100ms 或 500 条批量 Insert）。

2. 幂等设计（所有写接口必须幂等）：
   - 幂等 Key 生成规则：业务类型 + 业务ID（如 "PAY_ORDER_12345"）。
   - 实现方式：
     * Redis SETNX：写入前检查幂等 Key 是否存在。
     * 数据库唯一索引：INSERT 时利用唯一索引防重。
     * 状态机校验：UPDATE 时带状态条件（WHERE status = 'SETTLED'）。
   - 幂等 Key 过期时间：24 小时。

3. 数据对账兜底：
   - 定时任务（每小时）：检查"充电已结束但订单状态仍为 CHARGING"的异常订单 → 自动修正。
   - 定时任务（每日）：核对支付渠道流水与本地订单金额 → 差异告警。

【四、优雅启停与发布安全】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 优雅启动：
   - 预热阶段（Warmup）：启动后前 30s，逐步增加流量权重（从 0% → 100%），让 JIT 编译器和缓存预热。

   - 收到 SIGTERM 信号后：
     1. 从 Nacos 注销服务（停止接收新请求）。
     2. 等待 10s（让已发出的请求的负载均衡权重归零）。
     3. 关闭 HTTP 端口（返回 503 给新请求）。
     4. 等待进行中的请求处理完毕（最长 30s 超时）。

     * 任一阶段异常自动回滚。

   - 数据库变更兼容：
     * 遵循"扩展-收缩"模式：先加新列 → 双写 → 迁移数据 → 切读 → 切写 → 删旧列。
     * 禁止在发布期间执行 DDL（使用 gh-ost 在线 DDL）。

## 提示词 6：零信任安全架构与数据隐私防护

```text

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

   - 登录 IP 白名单（仅允许公司内网 IP 或 VPN IP）。
   - 会话管理：单点登录（同一账号最多 3 个活跃会话，超出踢掉最早的）。
   - 密码策略：最少 8 位，含大小写+数字+特殊字符，90 天强制更换，不能与前 5 次密码重复。
   
   模拟器 Web 端：
   - API Key + Secret 认证（HMAC-SHA256 签名）。
   - 请求签名算法：sign = HMAC-SHA256(secret, method + path + timestamp + nonce + body)。
   - 时间窗口校验：|server_time - timestamp| < 5 分钟。
   - Nonce 防重放：Redis 存储已使用的 Nonce（TTL 5 分钟）。

2. RBAC 权限模型（增强版）：
   - 权限粒度：菜单权限 + 按钮权限 + 数据权限 + 字段权限。
   - 数据权限：
     * 全部数据 / 本组织及下级 / 仅本组织 / 自定义组织 / 仅本人。
     * 使用 MyBatis-Plus 拦截器自动注入数据权限 SQL 条件。
   - 权限缓存：用户权限列表缓存在 Redis（Hash 结构），登录时加载，权限变更时主动失效。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、API 安全防护】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 接口签名（防篡改）：
   - 所有写操作（POST/PUT/DELETE）请求体签名。
   - 签名算法：SHA256(timestamp + nonce + sorted_params + body + app_secret)。
   - 网关层统一验签，验签失败返回 403。

2. 防重放攻击：
   - 每个请求携带 timestamp + nonce。
   - timestamp 有效期 5 分钟。
   - nonce 在 Redis 中记录（SETNX + TTL 5min），重复 nonce 拒绝。

3. 防刷与反爬虫：
   - 接口级频率限制（Sentinel 热点参数限流）。
   - 用户级频率限制（令牌桶算法，Redis + Lua 实现）：
     ```lua
     -- 令牌桶限流
     local key = KEYS[1]
     local capacity = tonumber(ARGV[1])  -- 桶容量
     local rate = tonumber(ARGV[2])      -- 每秒生成令牌数
     local now = tonumber(ARGV[3])       -- 当前时间戳(ms)
     
     local bucket = redis.call('HMGET', key, 'tokens', 'last_time')
     local tokens = tonumber(bucket[1]) or capacity
     local last_time = tonumber(bucket[2]) or now
     
     local delta = math.max(0, now - last_time)
     tokens = math.min(capacity, tokens + delta * rate / 1000)
     
     if tokens >= 1 then
         tokens = tokens - 1
         redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
         redis.call('EXPIRE', key, 60)
         return 1  -- 允许
     else
         return 0  -- 拒绝
     end
     ```
   - 验证码触发：同一操作连续失败 3 次 → 弹出图形验证码。
   - 行为分析：短时间内大量不同设备查询 → 疑似爬虫 → 加入黑名单。

4. 输入校验与注入防护：
   - 全局参数校验（Spring Validation + 自定义校验器）：
     * 所有字符串参数过滤 XSS 特殊字符（<, >, ', ", &, script）。
     * SQL 参数全部使用预编译（MyBatis #{} 占位符，严禁 ${}）。
     * 文件上传：白名单校验文件类型（Magic Number 校验，非仅扩展名）+ 文件大小限制 + 病毒扫描。
   - 富文本内容：使用 Jsoup 白名单过滤（仅允许安全标签）。

【三、数据加密与隐私保护】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 传输加密：
   - 全链路 HTTPS（TLS 1.3），禁用 TLS 1.0/1.1。
   - 证书管理：Let's Encrypt 自动续签 / 企业 OV 证书。
   - OCPP WebSocket：支持 WSS（TLS 加密），OCPP 2.0.1 强制 mTLS。
   - 内部微服务通信：使用 mTLS（通过 Istio Service Mesh 自动管理证书）。

2. 存储加密：

     * 用户手机号、身份证号、银行卡号、邮箱。

     * 使用 MyBatis-Plus TypeHandler 自动加解密（对业务代码透明）。
   ```java
   @TableField(typeHandler = AesEncryptTypeHandler.class)
   private String phone;  // 数据库中存储密文，Java 对象中为明文
   ```
   - 密钥管理：
     * 使用 HashiCorp Vault 或阿里云 KMS 管理加密密钥。
     * 密钥轮换：每 90 天自动轮换，旧密钥保留用于解密历史数据。
   - 数据库透明加密（TDE）：MySQL Enterprise TDE 或 Percona XtraDB 加密。

3. 数据脱敏：
   - 日志脱敏（Logback 自定义 Converter）：
     * 手机号：138****8888
     * 身份证：110***********1234
     * 银行卡：6222 **** **** 1234
     * 姓名：张*三
   - API 响应脱敏：
     * 使用 Jackson 自定义序列化器 @JsonSerialize(using = PhoneMaskSerializer.class)。
     * 不同角色看到不同脱敏级别（客服看中间4位脱敏，运维看全量）。
   - 数据库查询脱敏：
     * 导出功能自动脱敏（Excel 导出时手机号/身份证脱敏）。

4. 个人信息保护合规（PIPL/GDPR）：
   - 数据分类分级：
     * L1 公开数据：站点名称、地址、电价。
     * L2 内部数据：订单统计、设备参数。
     * L3 敏感数据：用户手机号、充电轨迹、支付信息。
     * L4 极敏感数据：身份证号、人脸数据。
   - 用户权利保障：
     * 数据导出：用户可申请导出所有个人数据（JSON/PDF 格式）。
     * 数据删除：用户注销时，匿名化处理个人数据（保留业务统计数据，删除可识别信息）。
     * 授权管理：用户可查看/撤回已授权的数据使用范围。
   - 数据生命周期管理：
     * 用户注销后：30 天冷静期 → 匿名化处理 → 保留统计价值。
     * 订单数据：保留 3 年（税务合规）→ 归档到冷存储 → 7 年后销毁。
     * 通信日志：保留 6 个月 → 自动清理。

1. 操作审计日志（不可篡改）：
   - 记录所有写操作：操作人、操作时间、操作类型、目标资源、操作内容、IP、User-Agent。
   - 审计日志写入独立的审计数据库（只追加，不可修改/删除）。

   - 审计日志保留 3 年。

2. 异常行为检测：
   - 规则引擎实时检测：

   - 容器镜像扫描：Trivy 扫描 Docker 镜像漏洞。

要求：

- Session 固定攻击防护：登录成功后重新生成 Session ID。
- CSRF 防护：管理后台所有写操作携带 CSRF Token（SameSite Cookie + Token 双重校验）。

1. 告警规则（Prometheus Alertmanager）：

   - CPU > 80% 持续 5 分钟 → P2 告警。
   - 内存 > 90% 持续 3 分钟 → P1 告警。
   - 磁盘使用 > 85% → P2 告警。
   - Pod 重启 > 3 次/小时 → P1 告警。
   - 节点 NotReady → P0 告警。

   应用层告警：
   - 接口 P99 延迟 > 3s 持续 2 分钟 → P2 告警。
   - 错误率 > 5% 持续 1 分钟 → P1 告警。
   - QPS 突降 > 50%（环比）→ P1 告警。
   - JVM Full GC > 3 次/分钟 → P1 告警。
   - 数据库连接池使用率 > 90% → P2 告警。
   - Kafka Consumer Lag > 10000 → P2 告警。

   业务告警：
   - 设备离线数突增 > 20%（环比）→ P1 告警。
   - 支付成功率 < 95% 持续 5 分钟 → P0 告警。
   - 1 分钟内充电启动失败 > 50 次 → P1 告警。
   - 单用户退款 > 5 次/小时 → P2 告警（风控）。

2. 告警路由与通知：

   - P1 告警：短信 + 企业微信/钉钉 + 邮件（15 分钟内响应）。
   - P2 告警：企业微信/钉钉 + 邮件（1 小时内响应）。
   - P3 告警：邮件（24 小时内处理）。
   - 告警升级：P0 告警 15 分钟未确认 → 自动升级到上级主管。
   - 告警抑制：同一根因的告警合并（如 MySQL 宕机导致多个服务报错，只发一条根因告警）。
   - 告警静默：维护窗口期间静默特定告警。
