## 提示词 1：用户认证体系与账户安全中心

> **定位**：用户端的"安全大门"。支撑微信一键登录、手机号登录、多端会话管理、账户安全、车辆绑定等用户基础能力。

```text
请生成充电平台"用户端后端"的"用户认证体系与账户安全中心"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + Spring Security 6 + Redis Cluster + PostgreSQL + JWT (RS256)。
设计目标：登录接口 P99 < 200ms，支撑百万级日活用户，安全等级达到金融级。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、多渠道登录注册】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 微信生态登录（小程序 + 公众号）：

   A. 微信小程序一键登录（核心流程）：
   ```
   用户点击"微信登录" 
     → 小程序调用 wx.login() 获取 code
     → 小程序调用 wx.getUserProfile() 获取用户信息（可选）
     → 后端接收 code
     → 后端调用微信 API: code2Session(appId, secret, code)
     → 获取 openid + session_key + unionid
     → 查询用户是否已注册（通过 unionid 关联）
       → 已注册：生成 JWT Token，返回
       → 未注册：自动创建用户 → 生成 Token → 返回
     → 发放新人礼包（异步）
   ```
   
   B. 微信手机号快速登录（免验证码）：
   ```
   小程序调用 getPhoneNumber 按钮 → 获取 encryptedData
     → 后端调用微信 API 解密手机号
     → 绑定手机号到用户账号
     → 后续可直接用手机号登录
   ```

   C. 微信公众号 H5 登录：
   - OAuth2.0 网页授权 → 获取 openid → 关联或创建用户。

2. 手机号验证码登录（通用方案）：
   ```
   用户输入手机号 → 点击"获取验证码"
     → 频率限制：同一手机号 60 秒内只能发 1 次，每天最多 10 次
     → 同一 IP 每小时最多 20 次
     → 生成 6 位随机验证码
     → Redis 存储：key = "sms:code:{phone}", value = code, TTL = 5 分钟
     → 调用短信服务商 API（阿里云/腾讯云 SMS）
     → 发送短信
   
   用户输入验证码 → 点击"登录"
     → Redis 查询验证码，比对（比对后立即删除，防重放）
     → 验证失败次数 > 5 次 → 锁定该手机号 30 分钟
     → 验证成功 → 生成 JWT Token
   ```

3. Apple ID 登录（iOS 端合规要求）：
   - 调用 Apple Sign In SDK → 获取 identityToken。
   - 后端使用 Apple 公钥验证 JWT Token 签名。
   - 获取 Apple User ID → 关联或创建用户。

【二、Token 与会话管理】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. JWT Token 设计：
   - Header: { "alg": "RS256", "kid": "key-2024-07" }
   - Payload:
     ```json
     {
       "sub": "U12345",             // 用户ID
       "tid": "T001",              // 租户ID
       "ct": "app",                // 客户端类型: app/miniapp/h5
       "did": "device-fingerprint", // 设备指纹
       "ml": "GOLD",               // 会员等级
       "roles": ["USER"],          // 角色
       "iat": 1720857600,          // 签发时间
       "exp": 1720864800,          // 过期时间 (2h)
       "jti": "unique-token-id"    // Token唯一标识
     }
     ```
   - 签名：RS256 非对称签名（私钥签发，公钥验证，网关缓存公钥）。
   - 密钥轮换：每月轮换签名密钥，旧密钥保留 7 天用于验证。

2. Token 刷新机制（无感续期）：
   ```
   客户端每次请求检查 accessToken 是否即将过期（剩余 < 10 分钟）
     → 如果即将过期：
       - 客户端用 refreshToken 调用 /api/v1/auth/refresh
       - 后端验证 refreshToken 有效性（Redis 查询）
       - 签发新的 accessToken + refreshToken
       - 旧的 refreshToken 失效（Redis 删除）
     → 客户端在后台静默完成，用户无感知
   
   refreshToken 失效场景：
     → 30 天过期
     → 用户主动登出
     → 用户修改密码
     → 管理员强制下线
     → 检测到异常登录（异地登录）
   ```

3. 多端会话管理：
   - 同一用户最多同时 5 个活跃会话（App + 小程序 + H5）。
   - 超出时踢掉最早的会话（Redis 记录所有活跃会话）。
   - 用户可在"设置→安全中心"查看所有活跃会话并手动踢出。

4. Token 黑名单（紧急情况）：
   - 用户登出 → 将 Token JTI 加入 Redis 黑名单（TTL = Token 剩余有效期）。
   - 安全事件 → 批量将某用户所有 Token 加入黑名单。
   - 网关层每次请求校验黑名单（Bloom Filter 优化查询性能）。

【四、安全防护机制】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 设备指纹与风控：
   - 客户端采集设备指纹（DeviceID + IMEI哈希 + 屏幕分辨率 + 系统版本）。
   - 首次登录新设备 → 标记为可信设备。
   - 陌生设备登录 → 额外验证（短信验证码 + 图形验证码）。
   - 异常检测规则：
     * 同一账号 5 分钟内从不同城市登录 → 冻结账号 + 短信通知。
     * 同一设备频繁切换账号 → 标记风险设备。
     * 模拟器/Root设备检测 → 限制支付功能。

2. 接口安全：
   - 所有写操作接口签名（防篡改）：
     * sign = SHA256(timestamp + nonce + body + appSecret)。
     * 网关层验签，timestamp 有效期 5 分钟。
   - 防重放攻击：nonce 唯一性校验（Redis SETNX）。
   - 图形验证码触发：
     * 登录失败 3 次 → 弹出滑块验证码。
     * 发送短信前 → 弹出图形验证码。
     * 频繁操作 → 弹出验证码。

3. 数据脱敏：
   - 手机号：138****8888（API 返回脱敏，数据库加密存储）。
   - 身份证：110***********1234。
   - 银行卡：6222 **** **** 1234。

要求：
- 登录接口 P99 < 200ms（微信 code2Session 异步化 + 本地缓存微信公钥）。
- Token 校验在网关层完成（< 5ms），服务层无需重复校验。
- 短信验证码使用 Redis 原子操作（防并发验证漏洞）。
- 新用户注册后异步触发：发放新人礼包、创建用户画像、记录注册来源。
- 密码存储使用 bcrypt（cost = 12）。
- 所有认证相关操作记录审计日志（登录成功/失败/登出/密码修改）。

## 提示词 2：站点发现、智能推荐与实时状态服务

> **定位**：用户打开 App/小程序后的核心体验——快速找到附近的可用充电站，看到实时空闲枪数、价格、距离，并获得智能推荐。

```text
请生成充电平台"用户端后端"的"站点发现、智能推荐与实时状态服务"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + Redis Cluster (Geo + Hash) + Elasticsearch + PostgreSQL (PostGIS) + Caffeine。
设计目标：附近站点查询 P99 < 100ms，支撑 10 万 QPS 高频查询。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、附近站点查询（核心高频接口）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. API 设计：
   ```
   GET /api/v1/app/stations/nearby
   参数：
     - lat, lng: 用户当前经纬度（必填）
     - radius: 搜索半径（米，默认 5000，最大 50000）
     - keyword: 搜索关键词（站点名称/地址）
     - filter: 筛选条件
       * deviceType: DC(快充)/AC(慢充)/ULTRA(超充)
       * status: AVAILABLE(有空闲枪)/ALL
       * hasParkingFree: 是否免费停车
       * minPower: 最小功率(kW)
       * maxPrice: 最高价格(元/度)
       * openNow: 是否营业中
     - sort: DISTANCE(距离)/PRICE(价格)/POWER(功率)/RATING(评分)
     - page, size: 分页
   ```

2. 多级查询架构（性能优化核心）：

   ```
   【L1 - Caffeine 本地缓存】
   - 缓存"热门城市中心坐标 → 附近站点列表"的预计算结果
   - TTL: 5 分钟
   - 命中率 > 30%（大量用户位置相近）
   
   【L2 - Redis Geo 空间查询】（主力查询层）
   - 使用 Redis GEO 数据结构存储所有在线站点：
     * Key: "stations:geo:{tenantId}"
     * Member: stationId
     * Score: 经纬度 (GEOADD)
   - 查询命令：
     GEORADIUS stations:geo:T001 lng lat radius m 
       WITHCOORD WITHDIST COUNT 50 ASC
   - 时间复杂度: O(N+log(M)), 极快
   
   【L3 - Elasticsearch 全文搜索】
   - 当用户输入关键词搜索时走 ES
   - 支持：站点名称、地址、标签模糊搜索
   - 支持：复合条件筛选 + 地理位置过滤 + 排序
   
   【L4 - PostgreSQL PostGIS 精确查询】
   - 仅在 Redis 不可用时降级使用
   - 支持复杂空间查询（多边形范围内站点）
   ```

3. 站点卡片数据组装（并行调用优化）：
   ```java
   // 每个站点卡片需要的数据来自多个服务，并行调用
   public List buildStationCards(List stationIds, GeoPoint userLoc) {
       
       // 并行调用 5 个数据源（虚拟线程）
       var stationFutures = stationIds.stream().collect(Collectors.toMap(
           id -> id,
           id -> CompletableFuture.supplyAsync(() -> stationCache.get(id), vtExecutor)
       ));
       
       var priceFuture = CompletableFuture.supplyAsync(
           () -> pricingService.getBatchPrices(stationIds), vtExecutor);
       
       var statusFuture = CompletableFuture.supplyAsync(
           () -> deviceStatusService.getBatchStatus(stationIds), vtExecutor);
       
       var distanceFuture = CompletableFuture.supplyAsync(
           () -> calculateDistances(stationIds, userLoc), vtExecutor);
       
       var ratingFuture = CompletableFuture.supplyAsync(
           () -> reviewService.getBatchRatings(stationIds), vtExecutor);
       
       // 等待所有数据就绪
       CompletableFuture.allOf(priceFuture, statusFuture, distanceFuture, ratingFuture).join();
       
       // 组装卡片数据
       return stationIds.stream().map(id -> StationCardVO.builder()
           .stationId(id)
           .name(stationFutures.get(id).join().getName())
           .distance(distanceFuture.join().get(id))        // "1.2km"
           .address(stationFutures.get(id).join().getAddress())
           .totalConnectors(statusFuture.join().getTotalCount(id))
           .availableConnectors(statusFuture.join().getAvailableCount(id))  // "空闲 3/8"
           .currentPrice(priceFuture.join().getCurrentPrice(id))  // 当前时段价格
           .priceTag(getPriceTag(priceFuture.join(), id))  // "谷时低价"
           .rating(ratingFuture.join().get(id))            // 4.8分
           .tags(buildTags(stationFutures.get(id).join())) // ["免费停车", "快充", "洗手间"]
           .photos(stationFutures.get(id).join().getPhotos())
           .openNow(isOpenNow(stationFutures.get(id).join()))
           .build()
       ).toList();
   }
   ```

   -- KEYS[2] = wallet:version:{userId}
   -- ARGV[1] = amount (分)
   -- ARGV[2] = orderId (用于幂等)
   
   -- 幂等检查
   local idempotentKey = "wallet:idempotent:" .. ARGV[2]
   if redis.call('EXISTS', idempotentKey) == 1 then
       return cjson.encode({status = "DUPLICATE", balance = redis.call('GET', KEYS[1])})
   end
   
   local balance = tonumber(redis.call('GET', KEYS[1]) or "0")
   local amount = tonumber(ARGV[1])
   
   if balance >= amount then

       redis.call('INCR', KEYS[2])
       redis.call('SET', idempotentKey, "1", "EX", 86400) -- 24h 幂等窗口
       return cjson.encode({
