## 提示词 1：智能告警中心与规则引擎

> **定位**：运维系统的"眼睛和耳朵"。从百万级设备、订单、系统中实时采集异常信号，通过规则引擎和AI算法精准识别告警，分级分派。

```text
请生成充电平台"产品运维后端"的"智能告警中心与规则引擎"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + Drools/Flink CEP (复杂事件处理) + Kafka Streams + Redis + PostgreSQL + Elasticsearch。
设计目标：支撑百万级设备实时告警检测，告警延迟 < 3秒，误报率 < 5%，支持复杂事件关联分析。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、告警数据采集与标准化】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 多源告警数据接入（Kafka Topic 消费）：

   数据源1 - 设备告警（来自 ocpp-service）：
   - Topic: device.alert
   - 内容：设备上报的 StatusNotification(Faulted) + ErrorCode
   - 故障码映射表（OCPP ErrorCode → 标准告警类型）：
     * ConnectorLockFailure → 枪锁故障（P2）
     * GroundFailure → 接地故障（P1，高危）
     * OverCurrent → 过流告警（P1）
     * OverVoltage / UnderVoltage → 电压异常（P2）
     * WeakSignal → 通信信号弱（P3）
     * InternalError → 设备内部错误（P2）
     * PowerMeterFailure → 电表故障（P2）
     * TemperatureTooHigh → 过温告警（P1）

   数据源2 - 设备离线检测（来自 heartbeat-monitor）：
   - 心跳超时判定：超过 3 × HeartbeatInterval 未收到任何消息。
   - 分级：离线 10 分钟（P3 提醒）→ 离线 30 分钟（P2 告警）→ 离线 2 小时（P1 严重）。
   - 批量离线检测：同一站点 5 分钟内 > 3 台设备离线 → 升级为"站点级网络故障"（P1）。

   数据源3 - 充电业务异常（来自 charging-service / order-service）：
   - 充电启动失败（RemoteStart 返回 Rejected）。
   - 充电异常中断（非用户主动停止）。
   - 计量异常（功率突降为0但状态仍为Charging）。
   - 订单超时（充电停止后 30 分钟未收到 StopTransaction）。
   - 计费异常（计算金额与预估偏差 > 50%）。

   数据源4 - 支付异常（来自 payment-service）：
   - 扣款失败 / 重复扣款。
   - 对账差异超过阈值。
   - 退款异常。

   数据源5 - 系统运维告警（来自 Prometheus/基础设施监控）：
   - 服务器 CPU/内存/磁盘告警。
   - 数据库慢查询/连接池耗尽。
   - Kafka 消费延迟/堆积。
   - 证书即将过期。

   数据源6 - 用户投诉（来自客服系统）：
   - 用户在 App/小程序提交的投诉工单自动转为告警事件。

2. 告警事件标准化（AlertEvent 模型）：
   ```java
   public class AlertEvent {
       String alertId;           // 全局唯一ID (雪花算法)
       String alertType;         // 告警类型编码
       AlertLevel level;         // P0(致命)/P1(严重)/P2(一般)/P3(提醒)
       String sourceType;        // DEVICE/ORDER/PAYMENT/SYSTEM/USER
       String sourceId;          // 设备ID/订单ID/服务器IP
       String tenantId;          // 租户ID
       String stationId;         // 关联站点ID
       String title;             // 告警标题
       String description;       // 告警详细描述
       Map context; // 上下文数据（设备参数、订单信息等）
       Instant detectedAt;       // 检测时间
       String ruleId;            // 触发的规则ID
       AlertStatus status;       // NEW/ACKNOWLEDGED/PROCESSING/RESOLVED/CLOSED
   }
   ```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、告警规则引擎（核心）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 内置规则类型：

   A. 简单阈值规则（单条件触发）：
   - 示例："设备温度 > 85℃" → P1 告警。
   - 示例："设备离线 > 30 分钟" → P2 告警。
   - 实现：直接代码判断（高性能，无引擎开销）。

   B. 时间窗口规则（滑动窗口聚合）：
   - 示例："同一设备 1 小时内启动失败 ≥ 3 次" → P2 告警 + 自动标记设备为"疑似故障"。
   - 示例："同一站点 10 分钟内充电中断 ≥ 5 次" → P1 告警 + "站点供电异常"推断。
   - 实现：Kafka Streams 滑动窗口 / Flink CEP。
   ```java
   // Flink CEP 示例：检测设备频繁重启
   Pattern restartPattern = Pattern
       .begin("restart")
       .where(event -> event.getAlertType().equals("DEVICE_REBOOT"))
       .times(3).orMore()
       .within(Time.minutes(30));
   
   CEPatternStream patternStream = CEP.pattern(alertStream, restartPattern);
   patternStream.select((pattern, metadata) -> {
       // 生成"设备频繁重启"高级告警
       return AlertEvent.builder()
           .alertType("DEVICE_FREQUENT_REBOOT")
           .level(AlertLevel.P1)
           .title("设备30分钟内重启3次以上")
           .build();
   });
   ```

   C. 关联分析规则（跨源事件关联）：
   - 示例："设备告警(过温) + 同一站点多设备同时告警 + 天气预报高温" → 降级为 P2（环境因素）+ 建议"启动站点散热检查"。
   - 示例："充电中断 + 支付扣款失败 + 用户投诉" → 关联为同一事件链，只生成 1 个综合工单。
   - 实现：Flink CEP 多流 Join / 自研事件关联引擎。

   D. 趋势预测规则（基于历史数据的异常检测）：
   - 示例："设备充电功率持续下降（7天趋势线性回归斜率 < 阈值）" → P3 预警"设备性能劣化"。
   - 示例："站点日充电量连续 5 天同比下降 > 30%" → P2 告警"站点运营异常"。
   - 实现：定时任务 + 时序数据分析（TDengine SQL / Python 模型调用）。

   E. 自定义规则（用户可配置）：
   - 提供 Web UI 让运维管理员自定义告警规则（条件组合器）。
   - 规则 DSL 示例：
     ```
     WHEN device.temperature > 80 
       AND device.power > 50kW 
       AND time.hour BETWEEN 12 AND 15
       AND weather.temperature > 35
     THEN alert(level=P2, type="HIGH_TEMP_UNDER_LOAD", 
                action="CREATE_TICKET", assign_to="SITE_ADMIN")
     ```
   - 规则热加载：规则变更后 30 秒内生效（通过 Redis Pub/Sub 通知所有节点刷新规则）。

2. 告警去重与收敛：
   - 去重：同一设备 + 同一告警类型 + 5 分钟窗口内 → 合并为一条告警（次数 +1）。
   - 收敛（告警风暴抑制）：
     * 同一站点 1 分钟内产生 > 20 条告警 → 触发"告警风暴"模式。
     * 自动收敛为 1 条"站点级异常"汇总告警 + 附带子告警列表。
     * 收敛窗口内后续告警自动静默，不再推送通知。
   - 根因关联：
     * 如果"站点网络中断" → 自动关联该站点所有设备的"离线告警"为子告警。
     * 展示根因→影响的拓扑关系图。

3. 告警分级与自动分派：

   告警级别定义：
   | 级别 | 描述 | 响应时间 | 通知方式 | 自动动作 |
   |------|------|---------|---------|---------|
   | P0 | 致命（火灾/触电风险） | 立即 | 电话+短信+App推送+大屏弹窗 | 自动停机+创建紧急工单 |
   | P1 | 严重（设备不可用） | 15分钟 | 短信+App推送+企微/钉钉 | 自动创建工单 |
   | P2 | 一般（性能下降） | 2小时 | App推送+企微/钉钉 | 待分配工单池 |
   | P3 | 提醒（预防性） | 24小时 | App推送 | 记录观察 |

   自动分派规则：
   - 按设备归属：设备→站点→运维站→运维小组→值班人员。
   - 按技能匹配：电气故障 → 持有电工证的运维人员。
   - 按负载均衡：优先分配给当前待处理工单最少的运维人员。
   - 按地理位置：优先分配给距离最近且在线的运维人员。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、告警通知与升级】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 多渠道通知分发：
   - App 推送（极光/个推）：所有级别。
   - 短信（阿里云 SMS）：P0/P1 级别。
   - 电话语音（阿里云 Voice）：P0 级别。
   - 企业微信/钉钉/飞书 Webhook：P1/P2 级别，发送到运维群组。
   - 大屏弹窗：P0 级别在运营大屏强制弹窗 + 声音告警。
   - 邮件：P2/P3 级别日报汇总。

2. 告警升级机制（Escalation）：
   - 第 1 级：通知值班运维工程师（0 分钟）。
   - 第 2 级：15 分钟未确认 → 通知运维组长。
   - 第 3 级：30 分钟未确认 → 通知区域经理。
   - 第 4 级：1 小时未确认 → 通知运营总监 + CTO。
   - P0 告警跳过第 1 级，直接通知所有层级。

3. 告警静默与维护窗口：
   - 支持配置"静默规则"：某站点维护期间（如 2024-07-13 14:00~18:00），该站点所有告警静默（不推送通知，但记录日志）。
   - 全局静默：系统发布窗口期间静默非关键告警。

要求：
- 告警规则引擎必须支持热更新（新增/修改规则无需重启服务）。
- 所有告警事件存储到 Elasticsearch（便于全文检索和聚合分析），保留 1 年。
- 告警处理全生命周期记录（创建→确认→处理→解决→关闭），每次状态变更记录操作人和时间。
- 提供告警统计 Dashboard API：按类型/级别/站点/时间段统计告警数量、MTTA(平均确认时间)、MTTR(平均修复时间)。
- P0 告警触发时自动执行预定义的"应急预案"（如自动 RemoteStop 所有充电、自动断电指令下发）。
```

## 提示词 2：工单全生命周期管理引擎

> **定位**：运维系统的"手脚"。将告警转化为可追踪的工单，实现从创建→分派→处理→验收→关闭的全流程数字化管理。

```text
请生成充电平台"产品运维后端"的"工单全生命周期管理引擎"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + Flowable (BPMN 工作流引擎) + PostgreSQL + Redis + WebSocket。
设计目标：支撑日均 10000+ 工单流转，流程灵活可配置，全程可追溯。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、工单模型与分类】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 工单类型（WorkOrderType 枚举）：

   A. 故障维修工单（由告警自动创建或手动创建）：
   - 关联告警ID、设备ID、站点ID。
   - 故障描述 + 故障码 + 建议处理方案（知识库自动推荐）。

   B. 预防性维护工单（定时任务自动创建）：
   - 定期巡检工单（每周/每月）。
   - 设备保养工单（充电次数达到阈值）。
   - 滤网清洁/液冷液更换工单。

   C. 安装部署工单：
   - 新站点建设/新设备安装。
   - 设备搬迁/更换。

   D. 巡检工单：
   - 站点日常巡检（环境、安全、设备外观）。
   - 特殊巡检（台风/暴雨/高温后巡检）。

   E. 用户投诉工单：
   - 由客服系统转入。
   - 关联用户ID、订单ID。

   F. 整改工单：
   - 安全检查发现的隐患整改。
   - 政府/监管部门要求的整改。

2. 工单数据模型（WorkOrder）：
   ```java
   public class WorkOrder {
       Long id;                      // 工单ID
       String orderNo;               // 工单编号 (WO-20240713-0001)
       WorkOrderType type;           // 工单类型
       WorkOrderPriority priority;   // 紧急/高/中/低
       WorkOrderStatus status;       // 状态（见状态机）
       String tenantId;
       String stationId;
       String deviceId;
       String alertId;               // 关联告警
       String title;
       String description;
       String location;              // 工单地点
       GeoPoint geoLocation;         // 经纬度
       
       // 人员
       String creatorId;             // 创建人
       String assigneeId;            // 当前处理人
       String assigneeGroupId;       // 处理组
       String supervisorId;          // 监督人
       
       // 时间
       Instant createdAt;
       Instant assignedAt;
       Instant acceptedAt;           // 接单时间
       Instant arrivedAt;            // 到场时间（GPS 验证）
       Instant resolvedAt;           // 解决时间
       Instant closedAt;             // 关闭时间
       Instant deadline;             // 截止时间（SLA）
       
       // SLA
       Integer slaResponseMinutes;   // 响应 SLA（分钟）
       Integer slaResolveMinutes;    // 解决 SLA（分钟）
       Boolean slaBreached;          // 是否 SLA 违约
       
       // 处理记录
       List actions;    // 操作记录
       List comments;  // 沟通记录
       List attachments; // 附件（照片/视频）
       
       // 结果
       String resolution;            // 解决方案描述
       String rootCause;             // 根因分类
       List sparePartIds;    // 使用的备件
       BigDecimal cost;              // 维修成本
   }
   ```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、工单状态机（BPMN 工作流）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

使用 Flowable BPMN 引擎管理工单流转：

1. 标准故障维修工单流程：
   ```
   [创建工单] 
     → [自动分派] → (分派失败?) → [人工分配]
     → [待接单] → (超时未接?) → [自动升级分派]
     → [已接单] → 运维人员出发
     → [前往中] → (到达站点电子围栏?) → [已到场]
     → [处理中] → 运维人员进行维修
       → [需要备件?] → [申请备件] → [备件领取] → [继续处理]
       → [需要支援?] → [请求支援] → [专家远程协助/派人增援]
     → [提交结果] → 上传维修照片 + 填写处理记录
     → [待验收] → (自动验收 / 人工验收)
       → 验收通过 → [已解决]
       → 验收不通过 → [重新处理]
     → [已解决] → [待关闭]
     → (7天无复发?) → [已关闭]
     → (7天内复发?) → [重新打开]
   ```

2. 关键节点详细说明：

   【自动分派节点】（Service Task）：
   - 输入：工单类型、设备位置、紧急程度。
   - 算法：
     a. 过滤：筛选"在线"且"未请假"且"持有对应技能证书"的运维人员。
     b. 评分：距离得分(40%) + 负载得分(30%) + 技能得分(20%) + 历史评价得分(10%)。
     c. 选择得分最高的人员自动分派。
   - 分派失败（无可用人员）：转入人工分配池 + 通知运维主管。

   【SLA 计时器】（Boundary Timer Event）：
   - 每个关键节点设置 SLA 超时定时器：
     * 待接单超时：P1=15min, P2=2h, P3=24h → 自动升级。
     * 到场超时：P1=1h, P2=4h, P3=48h → 告警通知主管。
     * 解决超时：P1=4h, P2=24h, P3=72h → 升级通知经理。
   - SLA 即将到期前（剩余 20% 时间）发送预警通知。

   【到场验证节点】（GPS 电子围栏）：
   - 运维 App 持续上报 GPS 位置。
   - 当运维人员位置进入站点电子围栏（半径 200m）时自动触发"已到场"。
   - 防作弊：GPS 位置 + WiFi 指纹 + 蓝牙信标三重验证。

   【验收节点】：
   - 自动验收（设备维修类）：
     * 远程检测设备状态是否恢复正常（在线 + 无告警 + 充电测试通过）。
     * 自动下发 RemoteStart 测试充电 → 持续 30s → RemoteStop → 确认设备正常。
   - 人工验收（巡检/安装类）：
     * 推送验收任务给站长/运维主管。
     * 验收人查看维修照片/记录 → 通过/驳回。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、工单协作与沟通】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 工单内部沟通（类似工单评论区）：
   - 运维人员、主管、客服可在工单内发消息沟通。

## 提示词 3：设备远程诊断与预测性维护

> **定位**：从"被动维修"走向"主动预防"。通过设备运行数据的深度分析，提前发现潜在故障，减少停机时间。

```text
请生成充电平台"产品运维后端"的"设备远程诊断与预测性维护"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + TDengine (时序数据库) + Python (scikit-learn/TensorFlow, 独立模型服务) + Kafka + Redis。
设计目标：实现设备健康度评分、故障预测、远程诊断，将设备非计划停机减少 40%。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、设备健康度评估模型】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 健康度评分体系（0~100分）：

   维度1 - 通信健康（权重 20%）：
   - 心跳丢失率（近 7 天）：0% = 满分，>5% = 0 分。
   - 消息响应延迟 P99：<1s = 满分，>5s = 0 分。
   - 重连次数（近 7 天）：0次 = 满分，>10次 = 0 分。

   维度2 - 电气健康（权重 30%）：
   - 电压稳定性（标准差/均值）：<1% = 满分，>5% = 0 分。
   - 功率因数：>0.95 = 满分，<0.85 = 0 分。
   - 转换效率（输出/输入）：>95% = 满分，<90% = 0 分。
   - 温度异常频次：0 次 = 满分，>5 次 = 0 分。

   维度3 - 业务健康（权重 25%）：
   - 充电启动成功率：>99% = 满分，<95% = 0 分。
   - 充电中断率（非用户主动停止）：<1% = 满分，>5% = 0 分。
   - 平均充电功率/额定功率比：>80% = 满分，<60% = 0 分。

   维度4 - 物理健康（权重 25%）：
   - 枪头插拔次数/设计寿命比：<50% = 满分，>90% = 0 分。
   - 累计运行时长/设计寿命比：<60% = 满分，>90% = 0 分。
   - 告警频次（近 30 天）：0 次 = 满分，>10 次 = 0 分。
   - 维修次数（近 6 个月）：0 次 = 满分，>3 次 = 0 分。

2. 健康度计算与展示：
   - 每日凌晨定时任务计算所有设备健康度。
   - 结果写入 Redis（供实时查询）+ PostgreSQL（供趋势分析）。
   - 健康度分级：
     * 90~100：优秀（绿色）
     * 70~89：良好（蓝色）
     * 50~69：一般（黄色）→ 建议预防性维护
     * 30~49：较差（橙色）→ 建议尽快检修
     * 0~29：危险（红色）→ 建议立即停用
   - 健康度趋势图：近 30 天健康度变化曲线（下降趋势告警）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、预测性维护（Predictive Maintenance）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 故障预测模型（独立 Python 微服务，通过 gRPC 调用）：

   模型A - 充电模块寿命预测：
   - 输入特征：累计运行时长、平均工作温度、温度波动幅度、功率循环次数、输入电压波动。
   - 输出：预测剩余使用寿命（RUL, Remaining Useful Life），单位：小时。
   - 算法：LSTM (长短期记忆网络) + 迁移学习（用实验室加速老化数据预训练，用实际运行数据微调）。
   - 触发条件：当 RUL < 500 小时 → 生成"预防性更换"工单。

   模型B - 枪锁/连接器故障预测：
   - 输入特征：插拔次数、锁止/解锁成功率趋势、锁止电流波形特征、温度。
   - 输出：未来 7 天枪锁故障概率 (0%~100%)。
   - 算法：XGBoost + 特征工程。
   - 触发条件：概率 > 70% → 生成"检查/更换枪锁"工单。

   模型C - 通信模块故障预测：
   - 输入特征：信号强度(RSSI)趋势、丢包率趋势、重连频率、SIM卡使用时长。
   - 输出：未来 14 天通信中断概率。
   - 算法：Random Forest。
   - 触发条件：概率 > 60% → 生成"检查天线/SIM卡"工单。

   模型D - 电表精度漂移检测：
   - 输入特征：输入侧电量（电网侧）vs 输出侧电量（桩侧）的偏差趋势。
   - 输出：电表是否发生精度漂移（二分类）+ 漂移量估算。
   - 算法：统计过程控制 (SPC) + CUSUM 算法。
   - 触发条件：偏差 > 2% → 生成"电表校准"工单。

2. 模型训练与更新流水线：
   - 数据来源：历史故障工单 + 设备运行数据（TDengine） + 设备维修记录。
   - 训练频率：每月自动重训练（使用最新数据）。
   - 模型版本管理：MLflow 管理模型版本、A/B 测试。
   - 模型评估指标：Precision、Recall、F1-Score、AUC。
   - 模型上线流程：训练 → 评估 → 灰度发布（10% 设备使用新模型）→ 全量上线。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、远程诊断工具箱】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 远程诊断指令集（通过 OCPP 协议下发）：

   A. 设备信息读取：
   - GetConfiguration：获取设备所有配置参数。
   - GetDiagnostics：远程下载设备运行日志文件（URL 上传到 OSS）。
   - TriggerMessage(BootNotification)：强制设备重新上报设备信息。

   B. 设备状态深度检查：
   - 远程读取设备内部传感器数据（温度、湿度、烟感、水浸）。
   - 远程读取电表详细数据（各相电压/电流/功率/功率因数/谐波）。
   - 远程读取充电模块状态（各模块工作/故障状态、均流情况）。

   C. 远程测试：
   - 远程触发枪锁测试（解锁→锁止→检测反馈）。
   - 远程触发绝缘检测。
   - 远程触发风扇测试。
   - 远程触发指示灯/蜂鸣器测试。

   D. 远程修复：
   - ChangeConfiguration：修改配置参数（如调整充电功率限制）。
   - Reset(Soft)：软重启（不中断当前充电，等待充电结束后重启）。
   - Reset(Hard)：硬重启（立即重启，中断当前充电）。
   - ChangeAvailability：临时禁用/恢复设备。

2. 诊断工作流（引导式故障排查）：
   - 运维人员在 App 中选择设备 → 进入"远程诊断"。
   - 系统根据当前告警自动推荐诊断流程（决策树）：
     ```
     设备离线
     ├─ 检查信号强度 → 信号弱 → 建议更换天线/调整位置
     ├─ 检查 SIM 卡状态 → 欠费/过期 → 建议充值/更换 SIM
     ├─ 检查供电状态 → 断电 → 建议检查空开/线路
     └─ 远程重启 → 重启后恢复 → 软件异常 / 重启后仍离线 → 硬件故障
     ```
   - 每个诊断步骤可一键执行（自动下发对应 OCPP 指令）。
   - 诊断结果自动记录到工单。

3. 设备日志远程分析：
   - 远程拉取设备运行日志（GetDiagnostics）。
   - 自动解析日志，提取关键错误信息。
   - 与知识库中的"日志特征→故障原因"映射表匹配。
   - 生成诊断报告："根据日志分析，设备在 2024-07-12 15:30 出现 CAN 总线通信错误，可能原因为充电模块#3 的 CAN 接口松动"。

要求：
- 健康度计算必须高效（10 万台设备在 30 分钟内完成计算，使用批处理 + 并行计算）。
- 预测模型服务独立部署（Python FastAPI + gRPC），可独立扩缩容。
- 远程诊断操作全量记录审计日志（操作人、时间、指令、结果）。
- 高危远程操作（Hard Reset、固件升级）需二次确认 + 审批流程。
- 预测性维护生成的工单需标注"预测性"标签 + 预测置信度 + 预测依据。
- 设备健康度数据对接大屏展示（全站设备健康度热力图）。
```

   public class InspectionItem {
       String itemId;
       String category;        // 分类
       String name;            // 检查项名称
       String description;     // 检查说明
       InspectionType type;    // 检查类型
       // 检查类型：
       // PASS_FAIL  - 合格/不合格

1. 运维概览 Dashboard：

   - 告警热力图（站点地图 + 告警密度热力）。
   - SLA 达标率仪表盘。
   - 本月 Top 10 故障设备列表。

2. 设备可靠性分析：
   - 各设备型号 MTBF 排行（识别质量差的型号，指导采购决策）。
   - 各设备型号故障类型分布（识别高频故障类型，指导预防性维护策略）。
   - 设备寿命曲线（浴盆曲线分析：早期故障/偶发故障/磨损故障阶段识别）。
   - 站点故障热力图（识别故障高发站点，分析环境/电力等因素）。

3. 运维效率分析：
   - 各区域/团队工单处理效率对比。
   - 各时间段工单分布（识别高峰时段，优化排班）。
   - 运维人员工作量分布（识别过劳/空闲人员，优化人力配置）。
   - 巡检发现问题率趋势（评估巡检有效性）。

4. 成本分析：
   - 运维成本趋势（人工+备件+差旅，按月/按区域）。
   - 备件消耗排行（识别高消耗备件，优化采购策略）。
   - 单台设备运维成本 vs 营收分析（识别"亏损设备"，评估是否更换/退役）。
   - 预测性维护 ROI 分析（预测性维护投入 vs 避免的非计划停机损失）。

   - 输出：建议更换设备列表 + 预估更换成本 vs 继续维护成本对比。

2. 人力配置优化建议：
   - 基于工单密度热力图 + 交通距离矩阵，优化运维人员驻点分布。
   - 建议："XX 区域工单密度高但运维人员覆盖不足，建议增加 2 名运维人员"。

3. 备件库存优化建议：
   - 基于历史消耗数据 + 季节性趋势 + 设备部署计划，优化安全库存水平。
   - 建议："XX 备件近 3 个月消耗增加 50%，建议安全库存从 10 提升至 15"。

4. 月度运维报告自动生成：
   - 每月 1 日自动生成上月运维月报（PDF）：
     * 核心指标汇总（环比/同比变化）。
     * 重大故障复盘。
     * 优秀案例表彰。
     * 问题与改进建议。
     * 下月重点工作安排。

要求：
- 分析数据使用 ClickHouse（OLAP）加速复杂聚合查询（亿级数据秒级响应）。
- 数据从业务库（PostgreSQL）通过 ETL（Canal + Kafka Connect）同步到 ClickHouse。
- 报表支持定时生成 + 邮件推送（每周一发送上周运维周报给管理层）。
- 所有分析图表支持导出为 PNG/PDF。
- Dashboard 支持自定义布局（拖拽式配置）。

```text
【运维后端全局编码约束】

1. 事件驱动架构：
   - 告警、工单、巡检等核心业务使用事件驱动（Domain Event + Kafka）。
   - 告警创建 → 发布事件 → 工单服务消费创建工单 → 通知服务消费发送通知。
   - 解耦各模块，提升系统弹性。

2. 数据一致性：
   - 工单状态变更使用乐观锁（version 字段）。
   - 库存扣减使用 Redis + Lua 原子操作。
   - 跨服务操作使用 Saga 模式保证最终一致性。

3. 安全性：
   - 运维人员只能看到自己负责区域的设备和工单（数据权限隔离）。
   - 远程操控指令必须记录审计日志（不可篡改）。
   - 敏感操作（远程重启、固件升级）需操作密码 + 审批。
   - GPS 轨迹等隐私数据加密存储 + 保留期限限制。

4. 性能要求：
   - 告警检测延迟 < 3 秒（从事件产生到告警创建）。
   - 工单列表查询 < 500ms（Elasticsearch 支撑）。
   - 知识库搜索 < 200ms（向量检索 + ES 混合排序）。
   - 报表生成 < 10 秒（ClickHouse OLAP 加速）。

5. 可靠性：
   - 告警服务必须高可用（多副本 + 故障自动切换），宁可重复告警不可遗漏。
   - 工单数据每日备份 + 异地灾备。
   - 定时任务（巡检生成、健康度计算）使用分布式调度（XXL-Job），支持失败重试。

6. 可扩展性：
   - 告警规则引擎支持热更新（新增规则无需重启）。
   - 巡检模板支持灵活配置（无需代码变更）。
   - 知识库支持插件式扩展（如集成外部知识源）。
   - AI 模型支持热替换（新版本模型灰度上线）。
