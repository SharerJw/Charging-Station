## 提示词 1：模拟器核心引擎与 OCPP 客户端集群管理

> **定位**：模拟器的基础骨架。负责管理海量虚拟充电桩实例的生命周期，维护与真实后端 OCPP 网关的 WebSocket 长连接集群。

```text
请生成充电平台"产品模拟器后端"的"核心引擎与 OCPP 客户端集群管理"模块代码。
技术栈：Java 21 (虚拟线程) + Spring Boot 3.3 + Netty 4.1 (WebSocket Client) + Reactor + Redis Cluster + Kafka。
设计目标：单节点支撑 5 万+ 虚拟充电桩并发连接，集群支撑 100 万+ 设备模拟，连接建立速率 > 5000 CPS (Connections Per Second)。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、虚拟设备实例管理（Device Instance Manager）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 虚拟设备模型（VirtualChargePoint）：
   - 基础属性：deviceId, tenantId, 型号(vendor/model), 固件版本, 枪数(connectorCount)。
   - 运行状态：UNINITIALIZED, CONNECTING, CONNECTED, DISCONNECTED, FAULTED, DESTROYED。
   - 内存结构优化：
     * 使用享元模式（Flyweight Pattern）共享不可变数据（如设备型号物模型模板、默认配置参数）。
     * 每个实例仅保留可变状态（当前状态、SOC、计量值、Session信息），单实例内存占用控制在 < 2KB。
     * 使用 Off-Heap 内存（如 Netty DirectBuffer 或 Chronicle Map）存储百万级实例，避免 JVM GC 停顿。

2. 设备生命周期管理：
   - 批量创建：接收 API 请求（如"创建 10000 台 120kW 直流桩"），异步生成实例并注册到本地 DeviceRegistry。
   - 批量启动：按照配置的速率（如 500台/秒）发起 WebSocket 连接，避免瞬时并发冲垮后端网关（防雪崩）。
   - 优雅停止：发送 OCPP StatusNotification(Faulted/Unavailable) → 正常关闭 WebSocket → 清理内存。
   - 销毁：从 Registry 移除，释放资源。

3. 设备配置与参数模板：
   - 支持从后台拉取真实设备的配置模板（ConfigurationKey），初始化虚拟设备的参数（如 HeartbeatInterval, MeterValueSampleInterval）。
   - 支持自定义覆盖（如模拟特定固件版本的 Bug 行为）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、OCPP WebSocket 客户端集群（Netty Client）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Netty 客户端配置（极致连接数优化）：
   - 使用 NioEventLoopGroup（Worker 线程数 = CPU 核心数 × 2）。
   - 使用 Java 21 虚拟线程处理业务逻辑（避免阻塞 Netty IO 线程）。
   - TCP 参数调优（针对海量短/长连接）：
     * SO_REUSEADDR = true
     * TCP_NODELAY = true
     * SO_KEEPALIVE = true
     * 连接超时：connectTimeoutMillis = 5000ms
   - 端口耗尽问题解决：
     * 配置多个本地 IP 绑定（Multi-homing），轮询使用不同 IP 发起连接。
     * 调整操作系统内核参数（fs.file-max, ip_local_port_range, tcp_tw_reuse）。

2. WebSocket 握手与认证：
   - URL 构建：ws(s)://{gateway_host}/ws/ocpp/{tenantId}/{deviceId}
   - 协议头注入：Sec-WebSocket-Protocol = ocpp1.6 (或 ocpp2.0.1)
   - 认证方式支持：
     * HTTP Basic Auth (Authorization: Basic base64(user:pass))
     * Bearer Token (Authorization: Bearer xxx)
     * mTLS 双向证书认证（加载虚拟设备证书库）。

3. 连接池与重连机制：
   - 断线重连策略：指数退避（1s, 2s, 4s, 8s... 最大 60s）+ 随机抖动（Jitter）。
   - 网络分区模拟：支持 API 强制断开特定比例的设备连接（模拟基站故障/断网），并在指定时间后恢复重连。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、OCPP 协议栈实现（双向通信）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 消息路由与分发（MessageDispatcher）：
   - 上行消息（模拟器 → 平台）：
     * 主动上报：Heartbeat, BootNotification, StatusNotification, MeterValues。
     * 事务消息：StartTransaction, StopTransaction。
     * 响应消息：对平台下发指令的 CallResult / CallError。
   - 下行消息（平台 → 模拟器）：
     * 监听平台下发的 Call 消息（RemoteStart, RemoteStop, Reset, UnlockConnector 等）。
     * 根据 MessageTypeId(2) 解析 Action，路由到对应的 ActionHandler。

2. 消息序列化与性能优化：
   - 使用 Jackson Afterburner 或 Protobuf (若 OCPP 2.0.1) 进行极速 JSON 序列化。
   - 消息对象池化（Recycler）：复用 OCPP Message 对象，减少 GC。
   - 零拷贝发送：使用 Netty CompositeByteBuf 组装消息帧。

3. OCPP 核心 Action 模拟逻辑：
   - BootNotification：
     * 启动时发送，携带设备信息。
     * 处理平台返回的 Status (Accepted/Pending/Rejected) 和 HeartbeatInterval。
   - Heartbeat：
     * 根据 HeartbeatInterval 动态调度定时任务（使用 HashedWheelTimer 优化百万级定时器内存）。
     * 收到平台响应后更新本地时钟同步。
   - StatusNotification：
     * 状态变更时主动上报。
     * 支持模拟各种故障码（ConnectorLockFailure, GroundFailure, OverCurrent 等）。

要求：
- 所有虚拟设备的连接状态、收发报文必须异步写入 Kafka（Topic: simulator_device_traffic），供前端 Web 实时展示和事后分析。
- 提供 Prometheus Metrics：当前连接数、连接建立速率(CPS)、消息发送/接收 QPS、消息延迟分位数、重连次数。
- 支持 OCPP 1.6J 和 OCPP 2.0.1 双协议栈切换（通过配置指定）。
```

## 提示词 2：高逼真度车辆 BMS 模型与电气参数仿真

> **定位**：让模拟的充电数据“看起来像真的”。实现电动汽车电池管理系统（BMS）的充电曲线模拟，生成符合物理规律的实时电气参数。

```text
请生成充电平台"产品模拟器后端"的"车辆 BMS 模型与电气参数仿真"模块代码。
技术栈：Java 21 + 数学计算库 (Apache Commons Math) + Reactor。
设计目标：生成高度拟真、带随机波动的充电功率/SOC曲线，骗过平台后端的"异常数据检测"算法。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、电动汽车 BMS 模型库（Vehicle Profile）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 车型数据库（内置 20+ 主流新能源车型参数）：
   - 特斯拉 Model 3 (电池 60kWh, 最大接收功率 170kW, 400V 架构)
   - 比亚迪 汉 EV (电池 85kWh, 最大接收功率 120kW, 500V 架构)
   - 小鹏 G9 (电池 98kWh, 最大接收功率 430kW, 800V 碳化硅架构)
   - 理想 MEGA (电池 102kWh, 5C 超充, 最大接收功率 500kW+)
   - 宏光 MINI EV (电池 13kWh, 最大接收功率 7kW, 交流慢充)
   - 参数结构：batteryCapacity(kWh), maxChargePower(kW), nominalVoltage(V), maxVoltage(V), minVoltage(V), chargeCurveProfile(曲线特征枚举)。

2. 充电会话模型（ChargingSession）：
   - 每次"插枪"生成一个 Session，绑定一辆虚拟车。
   - 初始状态：initialSOC (随机 10%~30%), targetSOC (随机 80%~100%)。
   - 环境因素：ambientTemperature (影响充电功率，低温降额)。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、充电曲线生成引擎（核心算法）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

真实的直流快充曲线并非直线，而是复杂的非线性曲线。模拟器需实现以下阶段：

1. 握手与预充阶段（0 ~ 30秒）：
   - 绝缘检测、低压辅助上电。
   - 电压缓慢攀升至电池当前电压，电流极小（< 2A）。
   - 模拟数据：Voltage 缓慢上升，Current ≈ 0，Power ≈ 0。

2. 恒流充电阶段 CC (Constant Current) - 主力阶段 (SOC 10% ~ 80%)：
   - 特点：电流保持最大值（或接近最大），电压随 SOC 升高而线性/微非线性上升，功率持续攀升至峰值。
   - 算法：
     * 设定目标电流 I_target = MaxPower / CurrentVoltage。
     * 电压 V_current = V_min + (V_max - V_min) * (SOC / 100)。
     * 引入随机波动：I_actual = I_target * (1 + random(-0.02, 0.02))（模拟 BMS 微调）。

3. 恒压充电阶段 CV (Constant Voltage) - 涓流阶段 (SOC 80% ~ 100%)：
   - 特点：电压达到上限（如 400V/800V）保持不变，电流呈指数级衰减，功率快速下降。
   - 算法（指数衰减模型）：
     * V_current = V_max。
     * I_current = I_peak * exp(-k * (SOC - 80))，其中 k 为衰减系数（根据车型配置）。
     * 当电流衰减至 < 2A 时，判定为充满。

4. 800V 超充特殊曲线（如小鹏 G9 / 理想 MEGA）：
   - 特点：在 SOC 10%~70% 区间能维持极高的峰值功率（如 400kW+），随后快速下降。
   - 引入"热管理降额"模拟：如果模拟连续快充导致"电池温度"过高，主动降低峰值功率 20%。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、实时计量数据生成（MeterValues 仿真）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

根据上述曲线，按 MeterValueSampleInterval（如 15s 或 60s）生成 OCPP MeterValues 报文：

1. 核心采样字段生成逻辑：
   - Energy.Active.Import.Register (累计电量, Wh)：
     * 累加计算：Energy += (Power_kW * 1000 * interval_seconds) / 3600。
     * 加入电表精度误差（±0.5% 随机偏移）。
   - Power.Active.Import (实时功率, W)：
     * 直接取当前曲线计算点的 Power 值。
   - Voltage (电压, V)：
     * 取当前曲线计算点的 Voltage 值。
   - Current.Import (电流, A)：
     * I = Power / Voltage。
   - SoC (电池荷电状态, %)：
     * SOC_current = SOC_initial + (Energy_charged / batteryCapacity) * 100。
   - Temperature (电池/枪头温度, ℃)：
     * 基础温度 + 随功率和时间上升的增量 + 随机波动。

2. 数据"做旧"与"拟真"处理：
   - 避免数据过于完美（直线）：引入高斯白噪声（Gaussian Noise）。
   - 模拟传感器漂移：长时间充电后，电压/电流采样值出现微小系统性偏移。
   - 模拟通信丢包：按 0.1% 概率随机丢弃某次 MeterValues 上报（测试平台后端的插值补全能力）。

3. 交流慢充（7kW/11kW）特殊处理：
   - 功率基本恒定（如 6.8kW），无明显的 CC/CV 阶段。
   - 三相电流平衡度模拟（A/B/C 三相电流差异 < 1%）。

要求：
- 曲线计算必须高效（避免在高频定时器中使用复杂浮点运算），可预计算好"标准曲线查找表(Lookup Table)"，运行时查表+插值+加噪。
- 支持通过 API 动态注入异常电气数据（如：电压突降、电流过载、SOC 跳变），用于测试平台后端的告警规则。
- 生成的计量数据必须与最终 StopTransaction 上报的 meterStop 严格一致（积分校验，防止平台后端对账失败）。
```

## 提示词 3：场景编排引擎与自动化测试框架 (CI/CD 集成)

> **定位**：将零散的模拟动作组合成复杂的业务场景（剧本），并提供标准化 API 供自动化测试流水线调用，实现充电平台的持续集成验收。

```text
请生成充电平台"产品模拟器后端"的"场景编排引擎与自动化测试框架"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + 状态机 (Spring Statemachine) + JUnit 5 + REST Assured。
设计目标：支持可视化/代码化编排复杂充电场景，提供 gRPC/HTTP API 供 CI/CD 流水线调用，自动生成测试报告。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、场景编排引擎（Scenario Engine）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 剧本定义（Scenario DSL / JSON 格式）：
   支持通过 JSON 或 YAML 定义复杂的测试剧本。
   示例：`scenario_full_charge_with_interruption.json`
   ```json
   {
     "name": "正常充电中途断电重连",
     "devices": [{"id": "CP-001", "profile": "120kW_DC"}],
     "vehicles": [{"id": "EV-001", "profile": "Tesla_Model3", "initialSOC": 20}],
     "steps": [
       {"action": "CONNECT", "device": "CP-001", "vehicle": "EV-001"},
       {"action": "REMOTE_START", "device": "CP-001", "idTag": "USER-001"},
       {"action": "WAIT", "condition": "SOC_REACH", "value": 50},
       {"action": "INJECT_FAULT", "device": "CP-001", "fault": "POWER_LOSS"},
       {"action": "WAIT", "duration": "60s"},
       {"action": "RECOVER", "device": "CP-001"},
       {"action": "WAIT", "condition": "SOC_REACH", "value": 80},
       {"action": "REMOTE_STOP", "device": "CP-001"}
     ],
     "assertions": [
       {"check": "ORDER_STATUS", "expected": "COMPLETED"},
       {"check": "BILLING_AMOUNT", "operator": ">", "value": 0}
     ]
   }
   ```

2. 步骤执行器（Step Executors）：
   - DeviceActionExecutor：控制设备连接、断开、重启、状态变更。
   - ChargingActionExecutor：触发扫码/刷卡/即插即充、远程启停。
   - FaultInjectorExecutor：注入硬件故障（急停、过温、锁失败）、网络故障（延迟、丢包、断网）。
   - TimeTravelExecutor：时间加速（将 1 小时的充电过程压缩到 1 分钟内模拟完成，通过加速内部时钟和 MeterValues 上报频率实现）。

3. 并发场景编排：
   - 支持"同时启动 1000 个设备充电"的并发步骤（Parallel Step）。
   - 支持"随机 10% 设备掉线"的概率步骤（Probabilistic Step）。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、自动化测试集成 API（CI/CD 友好）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

提供标准化的 RESTful / gRPC API，供 Jenkins/GitLab CI 等流水线调用：

1. 测试任务管理 API：
   - POST /api/v1/tests/run：提交测试剧本，返回 taskId。
   - GET /api/v1/tests/{taskId}/status：查询执行进度（PENDING, RUNNING, PASSED, FAILED）。
   - GET /api/v1/tests/{taskId}/report：获取详细测试报告（JSON/JUnit XML 格式）。
   - POST /api/v1/tests/{taskId}/abort：强制终止测试。

2. 与平台后端的"闭环验证"机制：
   - 模拟器不仅负责"发数据"，还要负责"验结果"。
   - 验证方式：
     * 方式A（API 轮询）：模拟器调用平台后端的 Admin API 查询订单状态/告警记录，与预期对比。
     * 方式B（事件监听）：模拟器订阅平台的 Kafka Topic（如 order.settled），等待特定事件到达并校验内容。
     * 方式C（数据库直连）：测试环境下，模拟器直接查询平台 MySQL 验证数据落库正确性（最快，但耦合高）。

3. 内置标准验收测试集（Regression Test Suite）：
   - 核心链路测试：扫码启动→正常充电→正常停止→支付成功。
   - 异常链路测试：充电中拔枪、充电中设备断网重连、支付扣款失败。
   - 计费准确性测试：跨峰谷时段充电，验证分段计费金额精确到分。
   - 并发压测基准：1000并发启停，验证平台接口 P99 延迟 < 500ms。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、故障注入与混沌工程（Chaos Engineering）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 设备级故障注入：
   - 硬件故障：ConnectorLockFailure(枪锁失败), GroundFailure(接地故障), OverCurrent(过流), WeakSignal(信号弱)。
   - 行为异常：不上报 Heartbeat、MeterValues 乱序上报、StopTransaction 延迟上报、伪造大额电量。

2. 网络级故障注入（通过 Netty ChannelHandler 拦截）：
   - 延迟注入：所有上行/下行消息增加 500ms~2s 随机延迟。
   - 丢包注入：随机丢弃 5% 的 WebSocket 帧。
   - 乱序注入：缓存消息并打乱顺序发送（测试后端的幂等和排序处理）。
   - 断网模拟：直接 close Channel，并在指定时间后恢复。

要求：
- 场景引擎必须支持"时间加速"（Time Dilation），例如将 60 分钟的充电过程在 60 秒内模拟完（MeterValues 上报间隔从 60s 缩短为 1s，电量累加步长放大 60 倍）。
- 测试报告必须包含：执行的步骤、实际发送的 OCPP 报文快照、平台后端的响应、断言成功/失败详情、耗时统计。
- 提供 Swagger UI 方便手动触发和调试测试场景。

## 提示词 4：大规模并发压测调度与资源隔离

> **定位**：解决"单机无法模拟百万设备"的问题。设计分布式调度架构，实现压测任务的拆分、下发、资源隔离和防雪崩控制。

```text
请生成充电平台"产品模拟器后端"的"大规模并发压测调度与资源隔离"模块代码。
技术栈：Java 21 + Spring Boot 3.3 + Kubernetes + Redis + Kafka + Prometheus。
设计目标：支持跨多节点分布式调度百万级设备压测，精准控制加压速率，防止模拟器自身或目标平台崩溃。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【一、分布式压测调度器（Load Test Scheduler）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 压测任务拆分与分发：
   - 用户在 Web 端创建压测任务："模拟 10 万台设备，在 10 分钟内全部上线并启动充电"。
   - 调度器（Scheduler Node）根据当前存活的 Worker Node 数量（如 10 个），将任务均分：每个 Worker 负责 1 万台设备。
   - 任务下发：通过 Redis Pub/Sub 或 Kafka Topic 将子任务指令下发给各 Worker。

2. 精准加压速率控制（Ramp-up Control）：
   - 避免"瞬间 10 万连接"冲垮平台网关。
   - 支持配置加压模型：
     * 线性加压（Linear）：每秒增加 500 个连接，持续 200 秒。
     * 阶梯加压（Step）：每 1 分钟增加 1 万连接，保持 2 分钟，再增加 1 万。
     * 脉冲加压（Pulse）：瞬间打入 1 万连接，测试平台弹性扩容能力。
   - Worker 节点内部使用令牌桶（Token Bucket）或漏桶（Leaky Bucket）算法，严格控制 WebSocket 握手发起速率。

3. 资源隔离与自我保护：
   - Worker 节点自我保护：
     * 监控本地 CPU、内存、文件描述符(FD)使用率。
     * 当 FD 使用率 > 80% 或 CPU > 90% 时，自动暂停接收新任务，并向调度器报告"节点过载"。
   - 平台保护（防打死）：
     * 监控平台后端的响应延迟和错误率（通过平台暴露的 Prometheus 指标或模拟器自身的统计）。
     * 当平台 P99 延迟 > 5s 或 错误率 > 10% 时，触发"熔断保护"，自动降低加压速率或暂停压测，防止平台彻底崩溃导致测试无法继续。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【二、Worker 节点集群管理（K8s 编排）】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 无状态 Worker 设计：
   - 每个 Worker Pod 仅负责执行分配到的设备模拟任务，不存储全局状态。
   - 设备实例数据全部在内存中，Pod 重启后由调度器重新分配任务。

2. K8s HPA 自动扩缩容：
   - 根据 CPU 使用率或自定义指标（如 `simulator_active_connections`）自动扩容 Worker Pod。
   - 配置 KEDA (Kubernetes Event-driven Autoscaling)：当 Kafka 中压测任务队列有消息时，自动拉起 Worker。

3. 网络优化（K8s 层面）：
   - 使用 HostNetwork 模式运行 Worker Pod（避免 NAT 转换开销，提升连接建立速率）。
   - 配置 Node 级别的内核参数优化（sysctl）：
     * fs.file-max = 2097152
     * net.ipv4.ip_local_port_range = 1024 65535
     * net.ipv4.tcp_tw_reuse = 1
     * net.core.somaxconn = 65535

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、压测数据准备与清理】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 压测前置数据准备：
   - 自动调用平台 Admin API 批量创建测试站点、测试设备白名单、测试用户账号。
   - 生成测试所需的 RFID 卡号库、VIN 码库。

2. 压测后数据清理：
   - 压测结束后，自动调用平台 API 清理产生的脏数据（测试订单、测试告警、测试流水）。
   - 或者在平台后端开启"压测模式"（通过 Header 注入 `X-Test-Traffic: true`），平台自动将压测数据路由到独立的 Shadow DB（影子库），不影响生产数据。

要求：
- 压测过程必须实时可视化：在 Web 端展示当前连接数曲线、消息吞吐量曲线、平台响应延迟曲线（类似 JMeter 的实时监控）。
- 支持"流量录制与回放"：录制真实环境中的 OCPP 报文（脱敏后），在模拟器中按时间戳精准回放，用于复现生产环境的疑难 Bug。
- 压测结束后自动生成 PDF/HTML 性能测试报告（包含吞吐量、瓶颈分析、资源消耗图）。
```
